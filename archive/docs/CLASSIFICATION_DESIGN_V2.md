# Synapse 분류 정확도 개선 — V2 설계서

## 근본 목표

> 저장, 인출, 대화 전체에서 실제로 가치 있게 동작하되, 최소한의 리소스로.

---

## 단계별 문제 정의와 파인튜닝 근거

### 저장 (Save)

**입력:** 자연어 텍스트 (일반 대화, 공문서, 의학 기록)
**출력:** 노드/엣지 JSON → DB 저장
**E2B의 역할:** 텍스트 → 그래프 구조 변환

**현재 문제 (실험 결과):**
```
입력: "신규로 채용된 자는 최초로 근무를 개시한 날부터 3개월간을 수습기간으로 한다."
E2B 출력 노드: "신규로 채용된 자", "최초로 근무를 개시한 날"  ← 구문형
E2B 출력 라벨: "구분한다", "의거채용된"                       ← 동사형
기대 출력 노드: "직원", "수습기간", "3개월"
기대 출력 라벨: "기간", "포함"
```

**파인튜닝이 필요한 이유:**
- 공문서 법령체 문장 패턴 → 파인튜닝으로 명사형 노드/라벨 출력 학습
- 파인튜닝 없이 프롬프트 few-shot으로 해결하면 매 호출마다 긴 프롬프트 필요 → 로컬 E2B 컨텍스트 한계

**파인튜닝의 한계 — canonical name 분리:**

파인튜닝은 패턴을 학습하지 매핑 테이블을 외우지 않는다.

| canonical name 유형 | 해결 방법 |
|---------------------|-----------|
| **범용** ("허리디스크", "연차유급휴가", "고혈압") | 파인튜닝으로 학습 가능 |
| **개인** (특정 회사명, 개인 프로젝트명, 별칭) | 파인튜닝 불가 — 사용자마다 다름 |

개인 canonical name은 **런타임에 관련 노드 목록을 E2B 프롬프트에 주입**하는 방식으로 보완.

**개인 canonical name 주입 전략:**

개인 노드는 수천~수만 개로 성장한다. 전체 주입은 애초에 설계에 없다.
**항상 Kiwi 형태소 매칭으로 입력과 관련 있는 노드만 필터링해 주입.**

```python
def get_candidate_nodes(input_text: str, conn) -> list[str]:
    """입력 텍스트 명사와 부분 매칭되는 노드만 반환. 최대 50개.
    
    그래프 규모에 무관하게 항상 이 방식 사용. 전체 주입 없음.
    """
    morphs = kiwi_nouns(input_text)  # 명사 추출
    rows = conn.execute(
        "SELECT name, weight FROM nodes WHERE status='active' AND (" +
        " OR ".join(["name LIKE ?" for _ in morphs]) + ")",
        [f"%{m}%" for m in morphs]
    ).fetchall()
    # 정렬 기준: 매칭된 형태소 수(다수 우선) → weight 높은 순 → 노드명 짧은 순(구체)
    def rank(r):
        hit = sum(1 for m in morphs if m in r["name"])
        return (-hit, -r["weight"], len(r["name"]))
    return [r["name"] for r in sorted(rows, key=rank)][:50]
```

노드 수 증가에 따른 품질 저하 지점: 형태소 매칭 결과 50개를 초과하는 시점.
그 전까지는 이 방식으로 충분. 초과 후 대응은 형태소 매칭 정밀도 개선(FTS, 임베딩 유사도) 순서.

### 인출 (Retrieve)

**입력:** 자연어 질문
**출력:** BFS 시작 노드 → BFS 탐색 → 정제된 컨텍스트 엣지

**1단계: E2B 쿼리 분석**

**현재 문제:**
```
질문: "허리 아파"
무파인튜닝 추출: ["허리"]  ← 그래프에 없는 노드명 → BFS miss

질문: "기술 스택 뭐야?"
무파인튜닝 추출: ["기술"]  ← 허브 노드 → BFS 탐색 폭발
```

**파인튜닝이 필요한 이유:**
- 저장 시 canonical name으로 노드 생성 → 인출 시 같은 기준으로 추출해야 BFS 히트
- 허브("기술") vs 구체 노드("Docker") 구분 패턴 학습

**개인 canonical name 보완:** 저장과 동일하게 기존 노드 목록을 런타임 주입.

```
인출 프롬프트:
  "그래프 노드 목록: [조용희, 허리디스크, ㈜더나은, ...]
   질문에서 탐색 시작 노드를 추출하세요. 허브 노드(연결 많은 일반 개념)는 제외.
   질문: 허리 아파?"
  → ["허리디스크"]
```

**2단계: BFS 탐색**

start_nodes에서 넓게 탐색. 노이즈 허용.

**BFS 컨텍스트 크기 제한:**
- 제약 조건: E2B 컨텍스트 8K 토큰, 한국어 BPE 기준 엣지 텍스트 약 40~50토큰/개
- **상한: 80개 엣지** (80 × 50 = 4,000토큰 + 질문/프롬프트 ~1,000토큰 = 5,000토큰 이내)
- 이 상한은 그래프 규모가 아닌 E2B 컨텍스트 윈도우에서 고정된 값
- 트리밍 기준: start_nodes로부터의 BFS 거리 (가까울수록 우선)

**3단계: E2B 정제**

BFS 결과(≤80개 엣지) + 질문 → E2B가 관련 없는 엣지 제거.

**파인튜닝 필요한가?** → **불필요.** 단, 실험으로 검증 필요 (현재 미검증).

### 대화 (Chat)

**입력:** 정제된 컨텍스트 엣지 + 질문
**출력:** 자연어 답변

**파인튜닝 필요한가?** → **불필요.** 벤치마크 100% (단순 케이스 기준).
특정 도메인 특화(법령 해석, 의학 조언)가 필요해지면 재검토.

## 엣지 라벨 — 자유형 + 사후 정규화

### 고정 라벨 폐기 이유

사전에 모든 관계 유형을 정의하는 건 불가능. 모델이 억지로 가장 가까운 라벨로 매핑하면 의미 왜곡.

### 파인튜닝과 사후 정규화의 역할 분리

두 가지는 다른 문제를 해결한다:

| 도구 | 해결하는 문제 | 예시 |
|------|---------------|------|
| **파인튜닝** | 동사형 → 명사형 패턴 (형식 일관성) | "구분한다" → "종류" |
| **label_normalize.py** | 동의어 라벨 병합 (의미 일관성) | "기간" = "기간값" = "duration" → "기간" |

파인튜닝 후에도 동의어 라벨은 남을 수 있음 → label_normalize.py 필요.

### 사후 정규화 방법

```python
# 빈도 기반: 낮은 빈도 라벨을 임베딩 유사도로 가장 가까운 고빈도 라벨에 병합
# MERGE_THRESHOLD: 전체 distinct 라벨 수의 0.5% 또는 최소 5회 중 큰 값
# 예: 라벨 200종이면 max(200×0.005, 5) = 5회 미만이 병합 후보
# 라벨 1,000종이면 max(1000×0.005, 5) = 5회 미만
# → 고정 임계값(3) 금지. 데이터 규모에 따라 자동 조정.
def get_merge_threshold(conn) -> int:
    total = conn.execute("SELECT COUNT(DISTINCT label) FROM edges").fetchone()[0]
    return max(int(total * 0.005), 5)
```

**임베딩 모델:** `snunlp/KR-SBERT-V40K-klueNLI-augSTS` (한국어 검증됨)
- 영어 혼용 라벨에는 `intfloat/multilingual-e5-small` 병행 가능
- 임베딩 모델은 오프라인으로 실행 (런타임 의존성 없음)

**활성화 조건:** distinct 라벨 수 ≥ 200개 전까지 실행 금지.
그 전에 실행하면 정상 라벨도 병합될 위험 있음.

## 파이프라인

### 저장 경로

```
입력 텍스트
  ↓
[Kiwi]
  - 복합어 분리 방지 (seeds 파일 → Kiwi 사용자 사전 등록)
  - 조사 → 엣지 방향 힌트 추출
  - "나/저" → identity 노드 치환
  ↓
[파인튜닝 E2B]
  프롬프트: Kiwi 방향 힌트 + 기존 노드 목록 (개인 canonical name)
  출력: 노드/엣지 JSON (명사형 노드, 명사형 자유 라벨)
  doc_mode: 입력에서 자동 감지 → 조/항/호 앵커 인식 + seq 카운터
  ↓
[스키마 검증] (경량)
  - JSON 파싱 실패 처리
  - 빈 노드명 제거
  ↓
DB 저장
```

### 인출 경로

```
입력 질문
  ↓
[E2B — 1단계: 쿼리 분석]
  프롬프트: 기존 노드 목록 + 허브 구분 지침
  출력: start_nodes (canonical name 기준, 허브 제외)
  ↓
[BFS — 넓게 탐색]
  - 정방향 탐색, 동적 허브 threshold 차단
  - doc_mode: 최소 역방향 (공문서 앵커 역추적)
  - 최대 80개 엣지로 트리밍 (BFS 거리 기준)
  ↓
[E2B — 2단계: 정제]
  BFS 결과 (≤80개 엣지) + 질문 → 관련 엣지만 추림
  ↓
[대화 E2B]
  정제된 컨텍스트 → 자연어 답변
```

## 기존 설계 요소 — 유지/폐기/수정

### 유지

| 요소 | 이유 |
|------|------|
| **domain → 노드 전환** | 사람마다 다른 허브가 자연히 도출됨 |
| **동적 허브 threshold** | BFS 탐색 폭발 방지. 그래프 성장에 따라 자동 조정 |
| **문서 앵커 (조/항/호)** | 공문서 저장/인출 구조의 핵심 |
| **edges.seq** | 문서 항목 순서 보존. 공문서 인출 시 선순위 맥락 제공 |
| **상향 탐색 종료 규칙** | "제목" 엣지 도달 시 탐색 멈춤. 허브로 탈출 방지 |
| **조사 → 엣지 방향 (Kiwi)** | LLM 없이 로컬 처리 |
| **숫자+단위 처리 규칙** | "15일", "3개월" 등 의미 있는 수치 보존 |
| **filter_rules hub_node_id** | 도메인 필터링. domain 컬럼 삭제 후 대체 |
| **label_normalize.py** | 파인튜닝 후 남는 동의어 라벨 병합 |

### 폐기

| 요소 | 폐기 이유 |
|------|-----------|
| **seed_loader 런타임 (SEED_DICT/ALIASES/COMPONENTS)** | 범용 canonical name은 파인튜닝으로 학습. 개인 canonical name은 기존 노드 목록 런타임 주입으로 대체 |
| **prematch.py** | 파일 미존재 (기획만 됐고 미구현). seed match 역할은 파인튜닝으로, user node match 역할은 노드 목록 런타임 주입으로 커버됨 |
| **앙상블 3모드** | E2B 1단계(키워드) + BFS + E2B 2단계(정제)로 대체 |
| **양방향 BFS 전면 적용** | 허브 폭발 유발. doc_mode에서만 최소 적용 |
| **고정 라벨 강제** | 사전 정의 불가. 자유형 + 사후 정규화로 대체 |
| **Stanza 대명사 해소** | 발생 빈도 낮음. "나/저"는 identity 고정 매핑으로 충분 |

### 수정

| 요소 | 수정 내용 |
|------|-----------|
| **저장 프롬프트** | few-shot 제거 → 기존 노드 목록 주입 (컨텍스트 효율적) |
| **인출 프롬프트** | 앙상블 3모드 → 기존 노드 목록 주입 + 허브 구분 지침 |
| **BFS** | 넓게 탐색 + 80개 상한 트리밍. 정교한 Option C 불필요 |

## 오프라인 파인튜닝 파이프라인

**파인튜닝 대상:** 저장 + 인출 1단계 (같은 E2B, 같은 canonical name 기준)

**Claude API 사용 범위:**
- 파인튜닝 데이터 생성 시에만 사용 (오프라인, 1회성)
- 입력 소스: 공문서(취업규칙, 근로기준법) — 개인 데이터 아님 → 외부 전송 무방
- 런타임 저장/인출/대화에는 Claude API 미사용 (로컬 E2B만)

### 학습 데이터 1 — 저장

```
소스:
  - 공문서 PDF → 조항 단위 청킹
  - 의학 도메인 예시 (직접 생성)
  - 일반 텍스트 예시 (기존 training.jsonl)

정답 노드명 참조:
  - 공통사전 CSV (범용 표준용어)
  - seeds/legal.txt (법령 복합용어)
  - seeds/medical.txt (KCD 일상 질환)

파이프라인: Claude A(초안) → Claude B(검수) → save_train.jsonl

예상 비용: ~480개 청크 × 2회(A+B) ≈ 약 1,000회 API 호출
```

**Claude A→B 검수 신뢰도 한계와 대응:**

B도 Claude이므로 A의 체계적 오류(특정 패턴에서 구문형 노드 생성 등)는 B가 잡지 못할 수 있다.

대응:
1. **B는 생성이 아닌 체크리스트 검증만** — 동사형 노드 탐지, 라벨 형식 검사 등 명시적 룰 기반
2. **규칙 기반 사후 자동 검사 추가** — 정규식으로 동사형 어미(`-는, -된, -하다, -하는`) 포함 노드 자동 플래그
3. **5% 샘플 수동 검토** — 480개 중 24개를 사람이 확인 후 파인튜닝 진행

```python
VERB_PATTERN = re.compile(r'(하는|된|하다|는\s*자|하여|되어|할\s*수)')
def flag_bad_nodes(nodes: list[str]) -> list[str]:
    return [n for n in nodes if VERB_PATTERN.search(n)]
```

### 학습 데이터 2 — 인출 키워드 추출

```
소스:
  - 저장 학습 데이터의 정답 노드에서 역방향 생성
    → Claude: "이 노드를 찾으려는 자연어 질문을 만들어라"
  - 허브 구분 예시 포함 (허브 → 구체 노드 방향)

파이프라인: save_train.jsonl → Claude → retrieve_train.jsonl

포맷:
  {"prompt": "허리 요즘 어때?",     "completion": {"start_nodes": ["허리디스크"]}}
  {"prompt": "기술 스택 뭐야?",     "completion": {"start_nodes": ["Docker","React"]}}
```

```
save_train.jsonl + retrieve_train.jsonl
  → MLX LoRA (unsloth 환경)

     저장 태스크 구분: system prompt "저장 모드"
     인출 태스크 구분: system prompt "인출 모드"
```

## doc_mode 자동 감지

수동 지정 없이 입력 텍스트 패턴으로 자동 결정.

```python
DOC_MODE_PATTERN = re.compile(
    r'제\s*\d+\s*조'       # 제N조
    r'|[①②③④⑤⑥⑦⑧⑨⑩]'  # 원문자 항 번호
    r'|\d+\s*호\b'         # N호
    r'|제\s*\d+\s*항'      # 제N항
)

def detect_doc_mode(text: str) -> bool:
    return bool(DOC_MODE_PATTERN.search(text))
```

doc_mode=True일 때만: 역방향 BFS(앵커 역추적) + seq 카운터 활성화.
일반 텍스트에서 역방향 BFS는 비활성화(허브 폭발 방지).

## Phase 3-a 롤백 전략

마이그레이션은 기존 코드를 즉시 깨뜨린다. 롤백 절차를 미리 정해둔다.

```python
# 마이그레이션 전 자동 백업
import shutil, datetime
backup_path = f"{DB_PATH}.bak.{datetime.date.today()}"
shutil.copy2(DB_PATH, backup_path)   # SQLite 파일 복사 (journal flush 후)

# 마이그레이션 실행 + 검증
run_migration()
assert verify_schema()   # 스키마 확인
assert verify_data()     # 노드/엣지 수 비교 (before vs after)

# 롤백 필요 시
# shutil.copy2(backup_path, DB_PATH)
```

배포 순서: 백업 → 마이그레이션 → 검증 → 3-b 스크립트 교체. 검증 실패 시 백업 복원 후 중단.

## 숫자 처리 규칙

| 케이스 | 처리 | 예시 |
|--------|------|------|
| 단독 수량 숫자 | 버림 | `1`, `80`, `3` |
| 숫자 + 단위 | 노드 | `15일`, `3개월`, `80퍼센트` |
| 영문 + 숫자 | 노드 | `M4`, `v3`, `iPhone15` |
| 조항 번호 | 앵커 노드 | `35조`, `74조` |
| 항/호 번호 | 앵커 노드 | `①②`, `1호` |

단위: `년, 일, 월, 시, 분, 개, 개월, 주, 퍼센트, %, 명, 건, 회, 번`
앵커 단위 (수량 아님): `조, 항, 호, 장`
