# Synapse 설계 — 저장/인출/대화

## 근본 목표

> 저장, 인출, 대화 전체에서 실제로 가치 있게 동작하되, 최소한의 리소스로.

---

## 원칙

- 저장은 Kiwi가 형태소 분리. LLM은 모호성 해소와 상태 변경 감지에만 개입.
- 인출의 범위는 질문과 단계별로 수렴된 노드·엣지의 조합으로 알 수 있는 맥락으로 정한다.
- 노드가 쪼개져도 괜찮다. 엣지로 연결되면 탐색이 다 찾는다.
- 엣지 라벨은 조사/어미 그대로. 대화 LLM이 한국어로 읽는다. 라벨 없는 연결도 허용.
- 라벨 없는 엣지는 두 노드 이름만으로 LLM이 관계를 추론한다. 정보 손실 아님.
- 숫자도 노드다. 사람들은 저마다 좋아하는 숫자가 있고 숫자로 많은 의미를 표현하기도 한다.
- 시제 어미(었/겠/더)는 의도적으로 시간축 탐색 허브로 기능한다. "언제 아팠지?"처럼 시간 기반 질문은 이 허브를 통해 과거 사건 전체를 탐색한다.
- 도메인은 사전에 정의하지 않는다. 연결이 많아진 노드가 자연스럽게 허브가 되고, 허브가 도메인으로 기능한다.
- 개인 그래프와 조직 그래프는 별도다. 개인이 조직에 속하면 조직 그래프에 접근할 수 있고, 조직은 권한에 따라 개인 그래프의 일부에 접근한다.
  - 개인: "나" 노드는 만들지 않는다. 모든 정보는 암묵적으로 나의 것이다.
  - 조직: 주어가 누구냐가 핵심. 주어 없으면 질문. 조직 날리지 관리 특화 모드다.

## 저장

### Kiwi 형태소 → 노드/엣지 규칙

| 품사 | 분류 | 처리 |
|------|------|------|
| 명사 (일반/고유/의존) | 노드 | 그대로 |
| 숫자, 외래어 | 노드 | 그대로 |
| 동사, 형용사 | 노드 | 그대로 |
| 부사 | 노드 | LLM이 구체값으로 치환. 불가능하면 사용자 질문 |
| 부정부사 (안/못) | 노드 | 그대로 |
| 대명사 | 노드 | LLM이 실제 대상으로 치환. 불가능하면 사용자 질문. "나/저"는 노드 생성 안 함 |
| 조사 | 엣지 라벨 | 그대로 (으로, 에게, 에서 등) |
| 어미 (관형형/선어말/연결) | 엣지 라벨 | 그대로 (는, 었, 고 등). 시제 어미(었/겠/더)는 시간축 허브 노드가 됨 |
| 종결어미 (EF) | 버림 | 문장 끝맺음. 의미 없음 |
| 동사파생접사 (하/되/시) | 엣지 라벨 | 앞 명사와 뒤 노드를 연결 |

토큰 순서대로 체인:
```
노드 --(엣지)--> 노드 --(엣지)--> 노드 ...
```

### 예시

**"허리 아파"**
```
허리(명사) + 아프(형용사) + 어(EF→버림)
허리 --(없음)--> 아프
```

**"허리 다 나았어"**
```
허리(명사) + 다(부사) + 낫(동사) + 었(선어말어미) + 어(EF→버림)
허리 --(없음)--> 다
다 --(없음)--> 낫
낫 --(없음)--> 었
```

**"진천으로 이사했어"**
```
진천(명사) + 으로(조사) + 이사(명사) + 하(동사파생접사) + 었(선어말어미) + 어(EF→버림)
진천 --(으로)--> 이사
이사 --(하)--> 었
```

**"스타벅스 좋아"**
```
스타벅스(명사) + 좋(형용사) + 어(EF→버림)
스타벅스 --(없음)--> 좋
```

**"안 좋아"**
```
안(부정부사) + 좋(형용사) + 어(EF→버림)
안 --(없음)--> 좋
```

> 부정 표현 처리: "스타벅스 안 좋아"는 `스타벅스 --> 안 --> 좋` 체인으로 저장된다.
> 인출 시 스타벅스에서 BFS → `스타벅스 --> 안`, `안 --> 좋` 트리플이 함께 LLM에 전달된다.
> LLM이 체인 전체를 보고 "스타벅스를 안 좋아한다"는 맥락을 판단한다.

**"좋아하는 거 뭐야"** (인출 질문. 저장 아님)
```
좋아하(동사) + 는(관형형어미) + 거(의존명사) + 뭐(대명사) + 이(지정사) + 야(EF→버림)
좋아하 --(는)--> 거
```

### 모드 분기

| | 개인 모드 | 조직 모드 |
|---|---|---|
| 주어 생략 | 그대로 저장 | LLM이 사용자에게 질문 |
| "나" 노드 | 생성 안 함 | — |
| doc_mode | 거의 사용 안 함 | 주 사용 (취업규칙, 법령 등) |

### 파이프라인

```
입력 텍스트
  ↓
[Kiwi 형태소 분석]
  토큰 + 품사 태그 출력. 버리는 품사 없음 (EF 제외).
  ↓
[DB 사전 조회]
  Kiwi 토큰에서 명사 추출 → 해당 노드의 기존 트리플 전부 조회
  → LLM 컨텍스트에 포함 (상태 변경 감지에 필요)
  ↓
[LLM]  temperature=0, max_tokens=256
  1. 상태 변경 감지
     "나았어", "그만뒀어" 등 + 기존 트리플 → 관련 기존 엣지 inactive 처리
     예: "허리 나았어" 입력 → 기존 `허리 --> 아프` 트리플 확인 → inactive
  2. 대명사 치환
     지시대명사("이거", "그분") → 대상 추론. 불가능하면 사용자 질문
     "나/저" → 노드 생성 안 함 (개인 모드)
  2-a. 주어 확인 (조직 모드만)
     주어 없으면 → 사용자 질문
  3. 부사 구체화
     시간 부사("오늘", "내일") → 실제 날짜로 치환
     장소 부사("여기", "거기") → 실제 장소 추론. 불가능하면 사용자 질문
     정도 부사("많이", "자주") → 수치 추론. 불가능하면 사용자 질문
  4. 모호한 것 → 사용자 질문 반환 + "답변 후 저장이 재개됩니다" 안내
  ↓
[저장 재진입]
  사용자 답변 수신 → 모호성 해소 → 저장 파이프라인 재시작
  ↓
[doc_mode 자동 감지]
  정규식으로 문서 패턴 감지 → 앵커 노드 생성
  ↓
DB 저장 (트리플 체인)
  ↓
[LLM 별칭 제안]  (선택적, 새 노드 생성 시에만)
  새로 생성된 노드에 대해 LLM이 별칭 후보 제안
  예: "스타벅스" 노드 → ["스벅", "Starbucks"] → 자동 등록
  판단 불확실 시에만 사용자 질문
  → aliases 테이블에 저장
```

### doc_mode 자동 감지

```python
DOC_PATTERN = re.compile(
    r'제\s*\d+\s*조'        # 제N조
    r'|[①②③④⑤⑥⑦⑧⑨⑩]'   # 원문자 항
    r'|\d+\s*호\b'          # N호
    r'|제\s*\d+\s*항'       # 제N항
)
```

doc_mode=True이면:
- `제N조` → 앵커 노드
- `①②` / `제N항` → 하위 앵커 노드
- `N호` → 그 아래 앵커 노드
- 앵커 간 계층 엣지 + seq 카운터

예시:
```
취업규칙 --(없음)--> 35조
35조 --(없음)--> 연차유급휴가
35조 --(항, seq=1)--> 35조①
35조① --(없음)--> 80
80 --(없음)--> 퍼센트
35조① --(없음)--> 출근
35조① --(없음)--> 유급휴가
유급휴가 --(없음)--> 15
15 --(없음)--> 일
```

## 인출

### 파이프라인

```
질문
  ↓
[LLM 확장]  temperature=0, max_tokens=256
  질문 의도 해석 → 관련 노드 후보 생성 (1회)
  예: "언제 아팠지?" → ["감기", "허리", "두통", "아프", "낫"] 후보
  ↓
[DB 매칭]
  1. aliases 우선 조회: 후보 키워드 → aliases 테이블 → 매핑 노드 포함
  2. 노드명 직접 매칭
  3. 연결 트리플 전부 (양방향)
  ↓
[LLM 필터]  temperature=0, max_tokens=512
  트리플(source — label — target) 단위로 질문과 관련 있는지 판단
  관련 있음 → 통과. source/target을 다음 탐색 후보로
  판단 불확실 → 통과 (제외보다 포함이 안전)
  ↓
통과 노드 중 visited 아닌 것 있으면
  → 그 노드의 연결 트리플 전부 + LLM 필터 → 반복
없으면
  → 종료
  ↓
최종 엣지 컨텍스트 → 대화 LLM → 답변
```

응답 시간이 길 수 있다. 정확성 우선으로 대기 허용. 사용자에게 "탐색 중" 상태 안내 필요.

### 종료 보장

```python
visited = set()
while True:
    new_nodes = 이번 레이어 통과 노드 - visited
    if not new_nodes:
        break
    visited.update(new_nodes)
```

방문한 노드는 다시 탐색 안 함 → 그래프가 유한하므로 반드시 수렴.

## 대화

최종 컨텍스트(트리플 목록) + 질문 → LLM 자연어 답변.

컨텍스트에 없는 내용은 모른다고 답변.

---

## LLM 설정값

| 단계 | temperature | max_tokens | 이유 |
|------|-------------|------------|------|
| 저장 모호성 해석 | 0 | 256 | 오염 방지, 짧은 판단만 |
| 인출 확장 | 0 | 256 | 노드 후보 목록만 |
| 인출 필터 | 0 | 512 | 이진 판단 반복 |
| 대화 | 0.3~0.5 | 2048 | 자연스러운 답변 |

## Aliases

노드의 다른 이름/변형을 등록해두는 테이블. 인출 정확도를 높이는 핵심 장치.

```sql
aliases: alias(PK), node_id
```

### 역할

- **인출**: "스벅"으로 질문해도 "스타벅스" 노드를 찾아냄. LLM 호출 없이 DB 조회만으로 동작.
- **저장**: 동의어, 외래어 변형, 줄임말 등을 같은 노드로 수렴시킴.

### 생성 방식

| 방식 | 시점 | 설명 |
|------|------|------|
| LLM 자동 제안 | 새 노드 생성 직후 | 줄임말·외래어 변형·다국어 표기 등 단순 동의어 → 자동 등록. 판단 불확실한 경우만 사용자 질문 |
| 사용자 수동 등록 | 언제든 | "스벅은 스타벅스야" → aliases에 추가 |
| missing 기반 | 인출 실패 시 | 매칭 실패 키워드 → 사용자에게 "이 단어를 어떤 노드에 연결할까요?" |

### 우선순위 (인출 DB 매칭 순서)

```
1. aliases 조회 (정확 매칭)
2. 노드명 직접 매칭
3. 노드명 substring 매칭
```

## 개인/조직 그래프 관계

### 구조

```
개인 그래프 (Alice)          조직 그래프 (팀A)
  └── 노드/엣지              └── 노드/엣지
        │                          │
        └──── 개인이 조직에 속함 ────┘
              → Alice는 팀A 그래프 접근 가능
              → 팀A는 권한 범위 내 Alice 그래프 접근 가능
```

- 개인 그래프와 조직 그래프는 물리적으로 분리된 별도 그래프다.
- 그래프끼리 연결되는 접점은 허브 수준이 아니라 멤버십(소속 관계) 수준이다.
- 개인은 여러 조직에 동시에 속할 수 있다.

### 허브 → 도메인 부상

도메인은 생성하는 것이 아니라 관찰하는 것이다.

```
초기: 허리, 감기, 두통, 병원, 약 노드가 각각 존재
       ↓ 연결 축적
허브 감지: 특정 노드의 degree가 임계값 초과
       ↓
도메인 부상: 해당 노드가 그 클러스터의 대표 허브로 기능
```

- 임계값은 설정 가능. 기본값은 상대적 degree 상위 N%.
- 허브 노드는 UI에서 시각적으로 강조된다.
- 도메인이 부상하더라도 노드/엣지 구조는 변하지 않는다. 조회 편의를 위한 레이블일 뿐이다.

### 권한 모델

| 대상 | 기본값 | 설정 가능 범위 |
|------|--------|----------------|
| 개인 그래프 전체 | 비공개 | 조직별 접근 허용/차단 |
| 특정 허브 서브그래프 | 비공개 | 조직별 읽기 전용 허용 |
| 조직 그래프 | 멤버 전체 읽기 | 역할별 쓰기 권한 |

## LLM 역할 요약

| 단계 | LLM 역할 | 비고 |
|------|----------|------|
| 저장 | 상태변경 감지, 대명사/부사 구체화, 사용자 질문 생성 | 모호한 경우에만 호출 |
| 인출 확장 | 질문 의도 해석 → 노드 후보 생성 | 1회 |
| 인출 필터 | 트리플 단위 관련성 판단, 불확실 시 통과 | 레이어마다 호출 |
| 대화 | 자연어 답변 생성 | 1회 |
