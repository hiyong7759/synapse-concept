# Synapse — 지식 구조 개념 모델 v4

## 한 줄 정의

**인류 공통의 개념 사전 위에, 각자의 경험을 연결하는 플랫폼.**

---

## Synapse가 해결하는 문제

LLM 서비스마다 메모리가 따로 놀고, 있어도 멍청하게 적용한다.
맥북을 물어보는데 맥미니 기준으로 답하는 식.

Synapse는 사용자의 정보를 그래프로 구조화해두고,
**질문에 맞는 맥락만 골라서** 어떤 LLM 서비스에든 쓸 수 있는 프롬프트를 만들어준다.
처음에 맥락이 정확하게 들어가면, 그 세션 안에서는 LLM이 알아서 잘 유지한다.

렌즈가 필요 없는 질문은 없다. 코딩 질문도 환경·경험·도구에 맞는 답이 나오고,
일상 질문도 건강·선호·제약에 맞는 답이 나온다.
맥락이 들어가면 모든 답변이 더 정확해진다.
Synapse는 특정 케이스의 도구가 아니라 **모든 LLM 사용의 맥락 레이어**다.

## 핵심 구조

### 공용 노드 사전 + 개인 엣지

```
공용 노드 사전 (모든 사용자가 공유)
  = 인류 공통의 개념: React Native, 허리디스크, 맥미니, Docker...
  = 단어 + 도메인 라벨 + 별칭
  = 앱에 기본 4만 개 내장 (~2MB), 서버에 풀셋 50만 개 (~29MB)
  ★ 숫자는 시뮬레이션 결과이며 검증 필요.
  ★ granularity 기준(하나의 노드 = 무엇인가) 설계 필요.

개인 엣지 (각 사용자의 로컬에만 존재)
  = 공용 노드 사이의 연결: "맥미니 ── 집", "React ── Poomacy"
  = 각자의 경험, 판단, 맥락
  = 절대 서버에 올라가지 않음
```

사전은 공유, 연결은 개인. 같은 "React Native" 노드를 전 세계 사용자가 공유하지만,
누가 그걸 어떤 맥락에서 경험했는지는 각자의 엣지에만 담긴다.

### 노드 = 수평한 단어

모든 노드는 동등하다. 상하 관계 없음. 트리 구조 아님.

```
맥미니, M4, 집, 개발, React, 허리디스크
→ 전부 같은 레벨의 노드.
```

### 엣지 = 선 하나

```
맥미니 ──(위치)── 집
맥미니 ──(스펙)── M4
맥미니 ──(용도)── 개발

이게 전부. 노드 두 개를 선으로 잇고, 메모(label)를 붙인 것.
```

엣지 type은 3종:
- **link** — 두 노드가 관련 있음 (label로 성격 구분: "위치", "스펙", "용도", "원인")
- **similar** — 의미가 비슷한 노드 (괜찮음 ≈ 가능)
- **same** — 같은 것의 다른 표현 (RTX3080 = 3080)

### 노드의 속성

```
id          — 고유 식별자
name        — 표시 이름
domain      — 도메인 (장비, 기술, 건강, 위치, 용도, 프로젝트, 판단...)
status      — active | inactive | deleted
source      — user | ai
weight      — 프롬프트에 포함된 횟수
safety      — safety 노드 여부 + 규칙
confidence  — 확신도 (공용 사전 매칭은 불필요, 커스텀 노드만)
```

### 확신도 (Confidence)

공용 사전 매칭은 사전이 곧 검증이므로 확신도가 불필요하다.
커스텀 노드 생성 시에만 확신도를 부여한다.

```
confidence ≥ 0.8  → 자동 저장
0.5 ≤ confidence < 0.8  → 저장하되 "미확인" 마크, 사용자 확인 유도
confidence < 0.5  → 저장 안 함
```

## 두 가지 대화 모드

### 모드 A: 축적 대화 (Synapse 세션)

시작부터 구조화 의도가 있는 대화. 데이터 축적을 위한 전용 흐름.

```
Synapse 세션 시작 →
대화 진행 →
세션 종료 시 자동으로 노드/엣지 추출 제안 →
"이런 노드를 발견했어요" →
사용자 확인 후 저장
```

### 모드 B: 일반 대화

평소처럼 LLM과 대화한다. 구조화하지 않는다.

```
일반 대화 진행 →
끝. 그냥 대화로 끝남.

단, 사용자가 원하면 invoke로 사후 투입 가능:
"이 대화 시냅스에 저장해" →
대화에서 노드/엣지 추출 → 확인 후 저장
```

두 모드는 사용자가 명확히 인지할 수 있어야 한다.
구분이 이질감 없이 자연스러워야 하되, 어떤 모드인지는 알 수 있어야 한다.

## 두 가지 동작

Synapse가 하는 일은 두 가지뿐이다.

### 1. 그래프 구축 (정보를 넣는다)

```
흐름:
  축적 대화 종료 또는 사용자 invoke →
  LLM이 대화에서 단어 추출 →
  공용 사전에서 매칭 (LLM 불필요) →
  매칭 안 되는 것만 커스텀 노드 생성 (LLM 필요, 확신도 반환) →
  엣지 연결 →
  사용자 확인 후 저장

공용 노드 매칭률이 높을수록 LLM 호출이 줄어든다.
대부분의 구조화를 기계적으로 처리할 수 있다.
```

### 2. 맥락 프롬프트 생성 (정보를 꺼낸다)

```
흐름:
  사용자가 질문 입력 →
  키워드로 노드 매칭 →
  맥락 기반 그래프 탐색 (아래 "탐색 규칙" 참조) →
  safety 노드 강제 포함 →
  서브그래프를 프롬프트 텍스트로 조립 →
  LLM에 전달
```

## 교정 철학

### 처음부터 완벽할 필요 없다

사람의 기억도 완벽하지 않다. AI와 대화하면서 개념을 이해했다고 마스터한 것이 아니다.
중요한 것은 **틀렸을 때 고치기 쉬운 구조**.

```
기존 메모리:  부정확 → 어디가 틀린지 모름 → 방치 → 계속 짜증
Synapse:     부정확 → 결과물에서 발견 → 역추적 → 수정 → 점점 정확해짐
```

### 역추적 가능

프롬프트에 포함된 노드 목록을 사용자에게 보여준다.
결과물이 이상하면 노드 목록에서 원인을 찾아 수정할 수 있다.

### 교정 흐름

```
사용자: "Kubernetes는 잘못 들어간 거야"
→ 해당 노드 status: active → deleted (또는 inactive)
→ 연결된 엣지도 비활성
→ 비활성된 엣지의 반대쪽 노드가 고아(다른 연결 없음)면 사용자에게 알림
→ 다음 프롬프트에서 제외됨
```

### source 구분: 내가 한 것 vs AI가 말한 것

```
source: user — 사용자가 직접 말한 것
  → 프롬프트: "이 사용자는 FCM 연동 경험이 있다"

source: ai — AI가 응답에서 언급한 것
  → 프롬프트: "Docker에 대해 설명을 들은 적 있다"
```

AI한테 설명 들은 것이 내 실력으로 둔갑하지 않는다.

### 경험의 궤적을 저장한다

Synapse가 저장하는 것은 실력 레벨이 아니라 **경험의 맥락**.
수준 판단은 LLM에 맡긴다.

### 시행착오도 경험이다

실패한 시도, 배제한 선택지, 겪은 에러 — 전부 맥락에 포함.

```
FCM 노드에 연결된 경험들:
  ├─ "Spring Boot에서 패키지 분리 방식으로 연동" (source: user)
  ├─ "firebase-admin 의존성 충돌 경험" (source: user)
  ├─ "별도 서버 분리는 비효율로 판단, 배제" (source: user)
  └─ weight: 28

→ 6개월 후 같은 문제를 만나면, 이 시행착오가 프롬프트에 포함되어
  같은 삽질을 반복하지 않게 됨.
```

## 렌즈: 그래프 위의 포인터

그래프 원본은 건드리지 않고, **어떤 노드를 보여줄지만 결정**한다.

### LLM용 렌즈

질문 맥락 → 그래프 탐색 → 관련 서브그래프만 프롬프트로 조립.

### 사람용 렌즈 (확장)

구조 변경 없이 렌즈 테이블만 추가하면 됨.

```
"개발자 모임용" → [기술, 프로젝트] 포함, [건강, 위치] 제외
"운동 모임용"   → [운동] 포함, 허리디스크는 포함하되 다른 건강 정보 제외
"친한 형한테"   → 거의 전부 포함, 특정 노드만 제외
```

도메인 단위로 열고 닫되, 노드 단위로 오버라이드 가능.

→ **"이거 봐요" 한 방으로 나를 소개할 수 있는 구조.**

## 탐색 규칙: 맥락 기반 탐색

홉 제한을 두지 않는다. **질문의 맥락이 유지되는 한 계속 따라간다.**

### 원칙

```
- 탐색 깊이 제한 없음. 그래프가 유한하고 맥락이 이탈하면 자연히 멈춤.
- 도메인이 아니라 질문의 맥락으로 유효성을 판단.
- 판단 주체는 LLM. 로컬 규칙으로는 맥락 판단이 불가능.
- 토큰 비용은 무시할 수준. 노드 수백 개도 1M 컨텍스트에서 미미.
```

### 흐름

```
1. 질문 키워드로 시작 노드 매칭 (로컬)
2. 시작 노드에서 BFS로 연결된 모든 노드 탐색 (로컬, 그래프가 유한하므로 자연 종료)
3. safety 노드 강제 포함
4. 전체 서브그래프를 프롬프트로 조립하여 LLM에 전달
5. LLM이 질문 맥락에서 유효한 노드만 판단하여 답변에 반영
```

탐색은 로컬에서 전체를 하고, 필터링은 LLM이 한다.
노드 수백 개도 1M 컨텍스트에서 토큰 비용이 미미하므로,
로컬에서 잘라내는 것보다 LLM에 다 주고 판단을 맡기는 게 정확하다.

### 검증 결과

```
[테스트 1] "허리 때문에 할 수 있는 운동 추천해줘"
  시작: 허리디스크
  탐색: 허리디스크 → 데드리프트 ✓, 트랩바 ✓, 시디즈T80 ✗(의자), L4-L5 ✓
       데드리프트 → 재발 ✓ / 트랩바 → 허리부담적음 ✓
  결과: 허리디스크, L4-L5, 데드리프트, 재발, 트랩바, 허리부담적음 (6개)
  → 시디즈T80 정확히 제외. 운동 맥락에 의자 불필요.

[테스트 2] "맥북에서 Homebrew 설치가 안 돼"
  시작: 맥북
  탐색: 맥북 → macOS ✓, 회사 ✗(Homebrew와 무관)
  결과: 맥북, macOS (2개)
  → A기관→KWCAG2.2→반려 체인 진입 안 함.

[테스트 3] "로컬 LLM 돌리려는데 어떤 장비가 좋아?"
  시작: llama.cpp
  탐색: llama.cpp → RTX3080 ✓, 13B ✓
       RTX3080 → 윈도우 ✓, 게임 ✗ / 13B → 가능 ✓
  결과: llama.cpp, RTX3080, 13B, 윈도우, 가능 (5개)
  → 게임 제외.

[테스트 4] "개발 환경 정리하고 싶은데"
  시작: 개발
  탐색: 개발 → 맥미니 ✓, React Native ✓
       맥미니 → M4 ✓, 집 ✗, Docker ✓
       React Native → Poomacy ✓, Expo ✓, 2년 ✗
       → ... 개발 맥락 유지되는 한 계속 확장
  결과: 14개 노드 (개발 관련 전체)
  → 집, 경험기간 제외. 개발 환경 맥락에 불필요한 것만 빠짐.
```

### safety 예외

safety 태그 노드는 탐색 규칙과 무관하게 강제 포함.

```
허리디스크(건강) [safety]
→ 운동 관련 질문 시 반드시 프롬프트에 포함
→ LLM 맥락 판단에서 제외되더라도 강제 삽입
```

### 렌즈 vs 사람의 직접 입력 — 렌즈의 가치

```
[코딩 질문] "React에서 useEffect 무한루프 왜 나와?"
  사람: 그냥 질문만 던짐
  렌즈: React Native + Expo 환경, 2년 사용 → Expo hot reload 이슈 진단 가능

[환경 질문] "빌드가 안 돼"
  사람: "맥미니에서 빌드가 안 돼"
  렌즈: EAS Build + Expo + React Native → 되물을 필요 없이 바로 진단

[넓은 맥락] "개발 환경 정리하고 싶은데"
  사람: "맥미니에서 개발하는데 정리 좀 하고 싶어" (일부만 입력)
  렌즈: 14개 노드 전체 포함 → 사람이 기억 못하는 맥락까지 제공

[자기 서술] "이력서 써줘"
  사람: 기억나는 것만 30분 끙끙대며 나열
  렌즈: 전체 경험 그래프에서 자동 추출 → 누락 없음
```

렌즈가 필요 없는 질문은 없다. 차이의 크기만 다를 뿐,
맥락이 들어가면 모든 답변이 더 정확해진다.

## 보안

보안은 기능이 아니라 전제 조건이다.

- **모든 엣지(개인 데이터)는 로컬에만 존재.** 서버에 저장하지 않는다.
- **LLM에는 서브그래프 요약만 전달.** 전체 그래프가 노출되지 않는다.
- **LLM 제공업체 변경 시 사용자 재동의.**
- **백업:** 로컬 내보내기(무료) / E2E 암호화 클라우드(유료, 패스프레이즈 분실 시 복구 불가).

### 노출 원장 (Exposure Ledger)

LLM에 전달된 모든 이력을 기록한다.

```
exposure_log:
  node_id     — 노출된 노드
  direction   — "out" (LLM에 전달) | "in" (LLM이 생성)
  target      — "structuring" | "lens" | "intake"
  provider    — "anthropic" | "openai" | ...
  date        — 노출 일시
```

LLM 제공업체를 변경하면 기존 데이터를 새 제공업체에 전달하기 전에
**사용자 재동의**를 받는다. 제공업체가 바뀌면 새로운 노출이다.

### 단계별 보안 원칙

```
1단계 (Claude Code 스킬):
  로컬에서만 실행. Claude가 엣지 결정을 포함해도 보안 문제 없음.
  공유 범위는 사용자가 설정.

2단계 (MCP 서버) 전환 시:
  엣지 생성 로직을 로컬로 분리해야 함.
  외부 LLM 서비스에 엣지 구조가 전달되지 않도록 설계.
```

## 공용 노드 사전

### 서버 + 로컬 하이브리드

```
서버: 풀셋 50만 개
앱 설치 시: 기본 4만 개 다운로드
사용 중 매칭 안 되면: 서버 조회 → 있으면 로컬에 추가 → 없으면 커스텀 생성
```

서버 조회는 "이 단어가 사전에 있는지"만 확인. 사용자의 엣지는 노출되지 않음.

### 미결 사항

```
- 4만/50만 개 숫자는 시뮬레이션 결과이며 검증 필요.
- granularity 기준: "하나의 노드 = 무엇인가"를 정의해야 함.
  (React Native는 하나? React와 Native 분리? React Native Navigation은?)
- 사전 자동 성장 로직: 프라이버시 모순 미해결.
  1단계에서는 제외하고 로컬 온리로 운영.
```

## 저장 용량

```
노드 하나: ~200 bytes (단어 + 라벨 + 별칭)
대화 원문은 저장하지 않음. 결론(노드와 엣지)만 저장.

개인 그래프:
  보통 사용 1년   →  2.4 MB
  헤비 사용 10년  → 47.5 MB
  극단적 10년     → 94.4 MB

공용 사전:
  기본 4만 개     →  2 MB
  풀셋 50만 개    → 29 MB

→ 사진 10~20장 수준. 저장 공간 걱정 불필요.
```

## 그래프 정리

정리가 필요한 이유는 용량이 아니라 **탐색 품질**.
쓸모없는 노드가 많으면 관련 없는 게 딸려올 확률이 올라감.

```
자동 정리:
  1년간 프롬프트에 한 번도 안 쓰인 노드 → inactive 처리
  source: ai + weight: 1인 노드 → inactive 처리

사용자 정리:
  "6개월간 안 쓰인 노드가 47개 있어요. 정리할까요?"
  → 목록 보여주고 유지/삭제 선택

삭제가 아니라 inactive 처리가 기본.
20~30년 전 경험도 갑자기 필요해질 수 있으므로.
inactive 노드는 탐색 대상에서 빠지지만 그래프에는 남아 있음.
```

## 1단계 구현: Claude Code 스킬

첫 타겟은 개발자. MCP를 쓸 줄 알고, 맥락 오염을 가장 심하게 겪는 사람들.

### 구조

```
synapse/
  skills/
    context/SKILL.md    — 맥락 프롬프트 생성 (질문 → 그래프 탐색 → 답변 반영)
    save/SKILL.md       — 대화에서 노드/엣지 추출 → 저장
    list/SKILL.md       — 그래프 조회 (노드, 엣지, 도메인)
    update/SKILL.md     — 노드/엣지 수정, 비활성화, 삭제, 복원
  scripts/
    init_db.py          — SQLite DB 초기화
    check_onboarding.py — 온보딩 상태 확인 (완료 기준 판정)
    get_context.py      — BFS 전체 탐색 → 프롬프트 조립 (한국어 조사 처리 포함)
    add_nodes.py        — 노드/엣지 일괄 추가
    list_nodes.py       — 그래프 조회
    update_node.py      — 노드/엣지 수정/비활성화/삭제/복원
  synapse.db            — 로컬 SQLite (개인 그래프)
```

### 저장소 전략

```
1단계: SQLite + 인접 리스트 + Python BFS 탐색
  → 노드 수천 개 수준에서 ms 단위 응답. 충분.
  → 시작 노드에서 BFS로 연결된 전체 노드를 탐색. LLM이 필터링.

2단계 (MCP 서버 전환 시):
  → 같은 SQLite 파일 유지
  → 상주 프로세스이므로 시작 시 인메모리 그래프 로드로 전환
  → 탐색 로직만 변경. 마이그레이션 비용 0.
```

### 온보딩

```
온보딩 완료 기준 (단계별 시뮬레이션으로 검증):
  총 노드 15개 이상 + 도메인 4개 이상 + 3개 도메인에 각 2+노드
  → 이 시점에서 75% 개별화 달성
  → 핵심 발견: 노드 수보다 도메인 다양성이 개별화율을 결정

첫 사용 시:
  check_onboarding.py → status: "onboarding" →
  사용자에게 자기소개 정보 제안 (이력서, GitHub, 장비, 건강 등) →
  정보 입력할 때마다 진행 상황 안내 →
  기준 충족 시 "준비 완료" → 정상 맥락 생성으로 전환

온보딩을 강제하지 않는다.
대화하면서 자연스럽게 쌓을 수도 있고, 한 번에 몰아서 할 수도 있다.
```

### 동작

```
대화 시작 시:
  check_onboarding.py 실행 → 온보딩 미완이면 안내, 완료면 맥락 생성
  → get_context.py로 키워드 매칭 (한국어 조사 자동 제거)
  → BFS 전체 탐색 → safety 별도 전달 → LLM이 필터링

모드 A (축적 대화):
  Synapse 세션으로 대화 →
  세션 종료 시 자동 노드/엣지 추출 제안 → 확인 후 저장

모드 B (일반 대화):
  평소처럼 대화 →
  저장하고 싶으면: "이 대화 시냅스에 저장해" (invoke) →
  대화에서 단어 추출 → 공용 사전 매칭 → 커스텀 노드 생성 → 엣지 연결 →
  사용자 확인 후 저장

Claude Code 구독 모델 사용 → 별도 API 비용 없음.
```

### 1단계에서 제외하는 것

```
source 구분 (user / ai):
  엣지 label의 구체성이 경험 깊이를 이미 표현한다.
  "패키지 분리 방식 연동"이면 해본 것, "개념 접촉"이면 들은 것.
  1단계에서는 source 필드 없이 운영. 필요성이 확인되면 이후 추가.

공용 노드 사전:
  1단계는 모든 노드를 커스텀으로 생성한다.
  사용자 데이터가 쌓인 후 공통 패턴을 추출해 사전화 (bottom-up).
  사전 없이도 시스템은 동작한다. 매칭 효율만 떨어질 뿐.
```

### 확장 로드맵

```
1단계: Claude Code 스킬 + Python 스크립트 (프로토타이핑, 개발자 타겟)
2단계: MCP 서버 전환 (다른 MCP 클라이언트에서도 사용 가능)
  ★ 이 시점에서 엣지 생성 로직을 로컬로 분리
3단계: 앱 (React Native) + 사람용 렌즈 + 공용 사전 서버
```
