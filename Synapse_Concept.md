# Synapse — AI의 맥락 시냅스

## 한 줄 정의

**LLM의 "전체 히스토리 전부 주입" 구조를 뒤집어, 지금 이 질문에 공명하는 맥락만 선택적으로 연결하는 대화 레이어**

---

## 문제 인식

### 현재 AI 대화의 근본적 한계

사람의 뇌는 "인증 시스템 만들어줘"라고 하면 보안 경험은 활성화되고 오키나와 맛집 기억은 활성화되지 않는다. 관련 없는 건 연결하지 않으므로 잡음이 없다.

그러나 현재 AI는 컨텍스트 윈도우에 들어간 모든 정보를 동일한 무게로 참조한다. Memory 항목, 대화 히스토리, System Prompt — 관련 있든 없든 전부 영향을 미친다. **선택적 비활성화가 없다.**

### 구체적 문제 3가지

1. **정적 컨텍스트 주입** — Memory나 System Prompt(CLAUDE.md 등)가 주제와 무관하게 항상 통째로 들어감
2. **세션 내 컨텍스트 오염** — 대화 초반의 배경정보가 주제 전환 후에도 계속 영향을 미침
3. **대화 히스토리 누적** — 이전 주제의 대화가 그대로 컨텍스트 윈도우를 차지하면서 노이즈가 됨

### 사용자가 겪는 현실적 고통

- 여러 섹션을 비슷한 패턴으로 구현할 때, 앞 섹션은 잘 되는데 뒤 섹션은 개판이 됨
- 새 세션을 열면 패턴이 확립되기 전이라 구현 패턴 자체가 달라짐
- 어디까지 다시 주입해야 할지 판단이 어려움
- "잘 되던 시점"으로 돌아가고 싶지만 방법이 없음
- 한 단어가 맥락 전체를 뒤집어버리기도 함
- 정형화된 내용이 무조건 주입되어 의미 없이 결과에 영향을 미침

### 더 근본적인 문제: 전체 주입이라는 전제 자체

현재 LLM의 대화 구조는 이렇다:

```
턴 61의 응답을 받으려면 → [턴1][턴2][턴3]...[턴60] + [턴61 질문] 전부를 인풋으로 보낸다
```

60턴 대화했으면 매번 60턴 전체를 인풋으로 넣는다. 관련 있든 없든. 이 "전체 주입"이라는 전제 자체가 모든 문제의 근원이다.

사람의 뇌는 이렇게 작동하지 않는다. 오늘 나눈 대화 60턴을 전부 떠올리면서 다음 말을 하는 사람은 없다. 지금 이 순간 필요한 기억만 활성화될 뿐이다.

**Synapse가 제안하는 구조:**

```
턴 61의 응답을 받으려면 → [턴3][턴17][턴42] + [턴61 질문] 관련된 것만 인풋
```

이것이 가능하면 세 가지가 동시에 해결된다:
- **컨텍스트 오염 제거** — 무관한 턴이 아예 들어가지 않으므로 영향 자체가 없음
- **컨텍스트 윈도우 절약** — 한계 안에서 진짜 필요한 정보만 채우므로 훨씬 깊은 대화 가능
- **비용 절감** — API 호출 시 토큰 수가 줄어듦

---

### "찾아내는 것"이 아니라 "연결되는 것"

논리적이고 해석을 잘하는 사람은 맥락을 "찾으러 가는" 것이 아니다. 이미 가진 지식과 경험이 주제에 반응하여 자동으로 연결된다. 필요한 시점에 필요한 맥락들이 자동으로 연결되면서 우선순위에 맞춰 조합되는 것이다.

Synapse는 AI에게 이 **선택적 연결 능력**을 부여한다. 뇌의 시냅스처럼, 지금 이 주제에 공명하는 맥락만 활성화되고 나머지는 조용히 있게 만든다.

### 이름의 의미: Synapse = 아키텍처

시냅스는 모든 뉴런이 항상 연결되어 있는 것이 아니라, 특정 자극에 반응하는 경로만 활성화되고 나머지는 비활성 상태로 유지하는 구조다. 자주 함께 활성화되는 경로는 연결이 강해지고, 안 쓰이는 경로는 약해진다.

Synapse라는 이름은 비유가 아니라 아키텍처 그 자체다:
- 모든 대화 턴을 항상 넣는 게 아니라, **지금 자극(사용자 질문)에 반응하는 턴만 활성화**
- 자주 함께 효과적이었던 맥락 조합은 **연결 강화** (레시피로 축적)
- 결과를 망쳤던 맥락은 **연결 약화** (디버거 데이터로 학습)

현재 AI에 없는 것이 시냅스다. LLM은 전체 주입만 할 줄 알지, 선택적 연결을 할 줄 모른다. Synapse가 그 선택적 연결 레이어가 된다.

## 핵심 동작: 연결 → 대화 → 보정 → 강화 루프

Synapse의 중심은 에디터가 아니라 **선택적 연결**이다. 에디터는 보정 수단일 뿐이다.

### 작동 루프

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  ① 자동 연결                                        │
│  사용자 질문 입력                                    │
│  → 전체 히스토리에서 관련 턴만 자동 선별             │
│  → messages 배열 조립                                │
│                    ↓                                 │
│  ② 대화                                             │
│  선택적으로 연결된 맥락으로 LLM 응답                 │
│  → 잘 되면 그대로 계속 (③ 건너뜀)                   │
│                    ↓                                 │
│  ③ 보정 (필요할 때만)                                │
│  응답이 이상할 때 사용자 개입                        │
│  → 🚩 플래그 → 이유 입력 → 원인 탐지 → 턴 교체     │
│                    ↓                                 │
│  ④ 연결 강화/약화 (자동)                             │
│  ②에서 잘 된 조합 → 연결 강화                       │
│  ③에서 문제였던 조합 → 연결 약화                    │
│  → ①의 자동 연결 정확도가 향상됨                    │
│                    │                                 │
│                    └──→ ① 로 돌아감                  │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**쓸수록 연결이 좋아진다.** 사람이 경험을 통해 "이건 관련 있고 저건 아니다"를 점점 잘 판단하게 되는 것과 같다. 비효율적이었거나 잘못된 연결은 다음에 활성화되지 않고, 효과적이었던 연결은 더 강하게 활성화된다.

### 각 단계 상세

#### ① 자동 연결 — Synapse의 핵심 동작

사용자가 질문을 입력하면, Synapse가 알아서 작동하는 기본 동작이다. 사용자는 아무것도 안 해도 "전체 주입"이 아닌 "선택적 연결"이 일어난다.

작동 방식:
- 모든 대화 턴은 벡터 임베딩으로 저장되어 있음
- 새 질문이 들어오면 벡터 유사도로 관련 턴 후보를 검색
- 연결 강도 데이터(④에서 축적된)를 반영하여 최종 선별
- 선별된 턴만으로 messages 배열을 조립하여 LLM에 전송

**핵심 난관: 단순 벡터 유사도만으로는 부족하다.**

"인증 구현해줘"로 검색하면 "인증"이 들어간 모든 과거 턴이 후보가 된다. 하지만 실패했던 시도, 다른 프로젝트의 인증, 맥락이 다른 인증 논의가 섞여 들어올 수 있다. 단순히 cos(θ) 유사도만 보면 이걸 구분할 수 없다.

**해결: 복합 스코어링**

최종 선별 점수 = 벡터 유사도 + 연결 가중치 + 시간 가중치

- **벡터 유사도 (cos similarity)**: 의미적 관련성의 기본 척도
- **연결 가중치 (success_count)**: ④단계에서 축적된 성공/실패 데이터. 이전에 이 턴이 좋은 결과를 만든 횟수가 반영됨. 같은 유사도라도 "검증된 턴"이 우선
- **시간 가중치 (recency bias)**: 유사도가 비슷하다면 최근 대화에 약간의 가중치를 더하여 현재 작업 흐름을 유지. 3개월 전 턴보다 어제 턴이 현재 맥락에 더 적합할 확률이 높음

이 복합 스코어가 쌓이는 학습 데이터와 함께 점점 정교해진다.

범위:
- 현재 세션 내 턴 (기본)
- 과거 세션의 턴 (세션 간 연결 — 확장)
- CLAUDE.md, System Prompt 등의 맥락 조각 (항목 단위 선택적 활성화)

이것만으로도 현재의 "전체 주입" 대비 압도적 개선이다. AI가 원래 가진 지능을 100% 발휘하게 해주는 레이어.

#### ② 대화 — 일반적인 AI 대화

선택적으로 연결된 맥락 위에서 LLM이 응답한다. 잘 되면 사용자는 그냥 대화를 이어가면 된다. 이 과정에서 "이 질문에 이 턴 조합이 사용되었고 결과가 좋았다"는 데이터가 자동으로 기록된다.

#### ③ 보정 — 사용자 개입 (필요할 때만)

응답이 이상할 때만 작동한다. 핵심 플로우:

1. 사용자가 이상한 응답에 🚩 플래그 버튼을 누름
2. "뭐가 이상했어요?" — 구체적 이유를 입력받음
3. 사용자의 판단 기준으로 이분 탐색(Git bisect 방식) 시작
4. 원인 턴 특정 → 영향 분석
5. **대안 제시** — 제거가 아니라, 원래 의도를 살리면서 오염을 방지하는 수정안 제안
6. 사용자 수락 시 턴 교체 → 재생성

보정 도구:
- 턴별 포함/제외 토글
- **턴 내부 부분 편집** — 턴 안의 특정 문장/단락만 수정·제거 (수정 시 벡터 자동 재생성)
- 턴 내용 전체 수정
- 스냅샷 저장/불러오기 (Git commit처럼)
- 브랜치 — 스냅샷에서 분기하여 다른 도메인 정보 주입 (Git branch처럼)

#### ④ 연결 강화/약화 — 학습 (자동)

②와 ③에서 발생한 데이터를 바탕으로 연결 가중치가 업데이트된다.

- **강화**: "인증 구현" 질문에 턴 3, 17, 42가 함께 선별되어 좋은 결과 → 이 조합의 연결 강도 증가
- **약화**: 턴 37이 원인으로 지목되어 교체됨 → 유사 주제에서 턴 37류의 연결 강도 감소
- **패턴 축적**: "예외 지시 시 범위를 명시하지 않으면 AI가 전체 방향으로 오해한다" → 프롬프트 코치 피드백으로 활용

이 데이터가 쌓일수록 ①의 자동 연결이 정확해진다. CLAUDE.md 30개 항목 중 "이 주제에는 항목 3, 7, 15가 효과적이고 항목 9는 해롭다"는 것을 Synapse가 학습하게 된다.

## 기능 분류 (루프 기준)

| 루프 단계 | 동작 | 사용자 경험 |
|---|---|---|
| **① 자동 연결** | 관련 턴만 선별하여 messages 조립 | 사용자는 아무것도 안 해도 됨 (기본 동작) |
| **② 대화** | 선택적 맥락 위에서 LLM 응답 | 일반 채팅과 동일 |
| **③ 보정** | 디버거 + 에디터로 원인 탐지·교정 | 이상할 때만 개입 |
| **④ 강화/약화** | 연결 가중치 자동 업데이트 | 사용자는 모름 (백그라운드) |

핵심은 ①과 ④다. 사용자가 체감하는 건 "쓸수록 AI가 맥락을 더 잘 잡는다"는 것이다.

## 기술 스택 (초안)

### 프론트엔드
- **Vite + React** (경량 시작) 또는 React Native Web (웹+모바일 동시)

### LLM 연결
- **Ollama** (localhost:11434) — 비용 0원, messages 배열 자유 조작
- Qwen, Llama 등 로컬 모델로 1차 스크리닝
- Claude API는 정밀 확인용으로 선택적 사용
- 어떤 LLM이든 동일한 messages 배열 구조이므로 모델 교체 자유

### 핵심 아키텍처: 듀얼 세션 + 선택적 연결

기존 LLM 대화 구조를 근본적으로 재설계한다.

**기존 구조 (전체 주입):**
```
사용자 질문 → [전체 히스토리 1~60턴] + [질문] → LLM → 응답
```

**Synapse 구조 (듀얼 세션):**

두 개의 분리된 세션이 존재한다.

```
┌─────────────────────────────────────────────────┐
│  세션 A: 의도 정제 세션 (가벼운 대화)            │
│                                                  │
│  사용자: "인증 구현해줘"                          │
│  Synapse: "어떤 프레임워크? 기존 패턴 따를 거야?" │
│  사용자: "Spring Security, JWT"                   │
│  Synapse: "공공기관 보안 요건도 포함?"             │
│  사용자: "ㅇㅇ"                                   │
│  Synapse: [구체화된 프롬프트 제시]                 │
│  사용자: "좋아" ← 프롬프트 승인                   │
└──────────────────────┬──────────────────────────┘
                       ↓ 승인 순간
┌──────────────────────┴──────────────────────────┐
│  벡터 검색 (구체화된 프롬프트 기준)               │
│  → 관련 대화 턴 선별                             │
│  → 관련 소스코드 (함수/클래스 단위) 선별          │
│  → 관련 CLAUDE.md 항목 선별                      │
│  → 연결 강도 가중치 반영                         │
└──────────────────────┬──────────────────────────┘
                       ↓
┌──────────────────────┴──────────────────────────┐
│  컨텍스트 프리뷰 (사용자 검토)                    │
│                                                  │
│  📝 프롬프트: "Spring Security 기반 JWT 인증..."  │
│                                                  │
│  🔗 연결된 맥락:                                  │
│  ✅ 턴 #17: 패턴 확립 대화     [펼치기] [편집]    │
│  ✅ 턴 #42: JWT 설정 논의      [펼치기] [편집]    │
│  ✅ auth.service.ts (함수 3개)  [펼치기] [편집]    │
│  ✅ CLAUDE.md #7: 코딩 컨벤션   [펼치기] [편집]    │
│  ⚠️ 턴 #33: 예외 처리 논의     [펼치기] [편집]    │
│                                                  │
│  각 항목: 내용 확인 / 토글(포함·제외)             │
│          부분 편집 / 연결 이유(유사도) 확인        │
│          수동 추가                                │
│                                                  │
│  [실행] [세션 A로 돌아가서 프롬프트 수정]         │
│                                                  │
│  ※ 이 단계에서의 사용자 편집 행위(항목 제거,      │
│    추가, 부분 수정)가 연결 강화/약화 데이터로     │
│    자동 축적되어 다음 번 자동 연결 정확도 향상    │
└──────────────────────┬──────────────────────────┘
                       ↓ 사용자 실행 승인
┌──────────────────────┴──────────────────────────┐
│  세션 B: 실행 세션 (클린 컨텍스트)               │
│                                                  │
│  [구체화된 프롬프트]                              │
│  + [검토/편집 완료된 관련 대화 턴들]              │
│  + [검토/편집 완료된 소스코드]                    │
│  + [검토/편집 완료된 설정 항목들]                 │
│  → 깨끗한 컨텍스트 윈도우에서 LLM 실행           │
└─────────────────────────────────────────────────┘
```

**핵심: 세션 A의 지저분한 대화는 세션 B에 하나도 들어가지 않는다.**

세션 B는 정제된 프롬프트 + 연결된 맥락만으로 시작하므로 완벽하게 클린하다. "앞 섹션 잘 됐는데 뒤 섹션 개판" 문제가 구조적으로 발생하지 않는다. 매번 클린 세션이면서도, 벡터 연결이 관련 패턴을 가져오므로 일관성은 유지된다.

#### 세션 A의 역할: 의도 정제 (쿼리 정제기)

AI가 혼자 프롬프트를 확장하는 것이 아니다. **사용자와 대화하며 의도를 명확하게 끌어내는 과정**이다. 세션 A는 일종의 "쿼리 정제기(Query Refiner)"로서, 벡터 검색의 정확도를 비약적으로 높이는 역할을 한다.

사용자의 단답형 "인증 구현해줘"로 벡터 검색하면 엉뚱한 턴이 섞일 수 있다. 하지만 세션 A를 거쳐 "Spring Security 기반, JWT 패턴, 공공기관 보안 요건 충족, Poomacy 기존 패턴 준수"로 구체화되면, 벡터 검색이 정확히 관련 있는 맥락만 찾아낸다.

- 사용자의 단답형 입력에서 빠진 정보를 질문으로 보충
- 모호한 부분을 구체화
- 최종적으로 구체화된 프롬프트를 제시하고 사용자가 검토·승인
- 가벼운 대화이므로 작고 빠른 로컬 모델로 충분

#### 컨텍스트 프리뷰: 실행 전 사용자 검토

세션 A에서 프롬프트가 승인되고 벡터 검색이 완료된 후, 세션 B 실행 전에 **컨텍스트 프리뷰** 단계가 존재한다. 사용자는 필요에 따라 이를 검토할 수 있다.

프리뷰에서 사용자가 할 수 있는 것:
- 각 항목을 펼쳐서 내용 확인
- 항목별 포함/제외 토글
- **턴 내부 부분 편집** — 턴 통째가 아니라 턴 안의 특정 문장/단락만 수정·제거 (아래 상세)
- 빠진 맥락 수동 추가
- 연결 이유 확인 (유사도 점수 등)
- 만족하면 실행, 아니면 세션 A로 돌아가서 프롬프트 수정

**프리뷰에서의 사용자 행위가 학습 데이터로 작동한다.** 특정 항목을 빼거나 추가하거나 부분 수정한 행위 자체가 "이 주제에 이 맥락은 불필요/필요/수정 필요"라는 신호다. 이것이 연결 강화/약화에 반영되어, 다음에는 프리뷰에서 수정할 것이 점점 줄어든다. 처음엔 프리뷰에서 많이 수정하지만, 쓸수록 "알아서 잘 골랐네, 바로 실행"이 된다.

#### 턴 내부 부분 편집

하나의 턴에는 여러 내용이 섞여 있을 수 있다:

```
턴 #17 (assistant):
"Spring Security 설정은 이렇게 합니다.     ← 핵심 (유지)
참고로 이전에 말씀하신 카드 레이아웃은     ← 노이즈 (수정/제거 가능)
이 방식과 잘 어울릴 수 있어요.
JWT 토큰 구조는 다음과 같습니다..."        ← 핵심 (유지)
```

턴 단위로만 on/off 하면 "좋은 내용이 있는데 노이즈도 섞여 있어서 넣기도 빼기도 애매한" 상황이 생긴다. 턴 내부 편집이 되면 좋은 부분만 살리고 노이즈만 걷어낼 수 있다.

프리뷰 또는 보정(③) 단계에서 텍스트 에디터처럼 턴 내용을 직접 수정 가능하다. 기술적으로는 messages 배열의 content 문자열을 편집하는 것이므로 복잡하지 않다.

**주의: 턴 내용이 수정되면 벡터 임베딩을 자동 재생성해야 한다.** 수정된 턴의 기존 벡터값은 무용지물이 되므로, 수정 완료 시점에 비동기로 해당 턴만 재임베딩(re-embedding)하는 파이프라인이 필수다.

재임베딩 파이프라인:
1. 사용자가 턴 내용 수정 완료
2. 수정 전 버전을 히스토리로 보존 (롤백 가능)
3. 수정된 텍스트로 벡터 임베딩 재생성 (비동기)
4. 벡터 DB에서 기존 벡터를 새 벡터로 교체
5. 연결 가중치는 유지 (내용 수정이지 새 턴이 아니므로)

단일 턴 임베딩은 로컬 CPU에서도 밀리초(ms) 단위로 끝나므로 사용자 체감 지연은 없다.

#### 세션 B의 역할: 클린 실행

승인된 프롬프트를 기반으로 벡터 검색 후, 사용자가 프리뷰에서 검토·편집한 맥락으로 깨끗한 컨텍스트 윈도우에서 실행한다.

- 세션 A의 대화 히스토리는 포함되지 않음
- 프리뷰에서 검토/편집 완료된 맥락만으로 구성 → 노이즈 제로
- 매번 새 컨텍스트 윈도우 → "세션"이라는 개념의 제약에서 자유로움
- 큰 모델이 필요하면 여기서만 사용 (비용 효율)

#### 벡터화 대상 (맥락 풀)

다양한 레벨의 맥락이 같은 벡터 풀에 존재해야 한다:

| 맥락 유형 | 청킹 단위 | 예시 |
|---|---|---|
| 대화 턴 | 턴 단위 (user+assistant 쌍) | 이전 대화에서 패턴 확립한 턴 |
| 소스코드 | 함수/클래스/블록 단위 | Poomacy의 인증 관련 코드 |
| 설정/규칙 | 항목 단위 | CLAUDE.md의 개별 규칙 |
| 도메인 지식 | 개념 단위 | 공공기관 보안 요건 문서 |

청킹 전략이 Synapse의 성패를 좌우한다. 너무 크면 불필요한 정보가 따라 들어오고, 너무 작으면 맥락이 깨진다.

#### 컨텍스트 윈도우에 대한 관점 전환

컨텍스트 윈도우가 크면 좋긴 하다. 하지만 이유가 다르다:

- **기존**: 대화 히스토리를 우겨넣기 위해 커야 함 → 낭비
- **Synapse**: 히스토리는 벡터 DB가 담당, 윈도우는 "이번 턴에 필요한 맥락을 알차게 채우는 공간" → 효율

매 인풋마다 새로운 클린 윈도우를 열어도 된다. 연속성은 벡터 연결이 보장하므로, 윈도우는 순수하게 작업 공간으로만 사용된다. 100턴째든 1000턴째든 인풋 크기가 거의 일정하다.

### 벡터 임베딩
- 로컬 임베딩 모델 (Ollama의 nomic-embed-text 등) — 비용 0원
- 각 대화 턴을 벡터로 변환하여 저장
- 새 질문이 들어오면 유사도 기반으로 관련 턴 검색

### 데이터 저장
- 대화 히스토리, 스냅샷, 태그, 레시피 → 로컬 저장 (SQLite)
- 벡터 임베딩 → 로컬 벡터 DB (ChromaDB 등)
- 맥락 영향도 추적 데이터 → SQLite

## 차별점

### 기존 도구와의 차이

기존 도구들은 **"좋은 프롬프트를 생성하는 것"**에 집중한다. LLM의 전체 주입 구조는 건드리지 않는다.

Synapse는 **LLM의 대화 구조 자체를 재설계하는 레이어**다:
- 전체 주입 → 선택적 연결
- 사전 생성 → 사후 교정 + 실시간 최적화
- 단발성 프롬프트 → 학습 루프로 축적

### 포지셔닝

| | 프롬프트 빌더 | RAG 도구 | Synapse |
|---|---|---|---|
| 대상 | 새 프롬프트 | 외부 문서 | 대화 히스토리 자체 |
| 시점 | 사전(事前) | 사전 | 사전 + 사후 + 실시간 |
| 구조 | 전체 주입 유지 | 전체 주입 유지 | **선택적 연결로 전환** |
| 학습 | 없음 | 없음 | 맥락 인과관계 축적 |

이런 포지션의 도구는 현재 존재하지 않는다.

## MVP 범위 (루프 기준)

### Phase 0: 온보딩 — 맥락 풀 초기 구축

벡터 DB가 비어있는 상태에서는 연결할 맥락이 없으므로 Synapse의 가치를 체감할 수 없다. 사용자가 첫 세션부터 "오, 내 상황을 알고 있네?"를 느끼려면 초기 데이터 구축이 필수다.

**문서 가져오기 (자동 벡터화)**:
- CLAUDE.md → 항목별 분리 후 벡터화
- 프로젝트 소스코드 → 함수/클래스/블록 단위 벡터화
- 기존 대화 로그 (JSON, JSONL 등) → 턴별 벡터화
- README, PRD, 기획서, 도메인 문서 → 청크별 벡터화

**대화로 보충 (심층 온보딩 인터뷰)**:

단순히 몇 가지 질문을 던지는 게 아니라, 사용자의 맥락을 심층적으로 파악하는 구조화된 인터뷰를 진행한다. 이 과정에서 수집된 정보는 중요도에 따라 분류되어 벡터 풀에 가중치가 다르게 저장된다.

핵심 파악 (필수 — 높은 가중치):
- AI 에이전트 사용 목적 — "AI로 주로 뭘 하려고 하세요?" (코딩, 문서 작성, 분석, 학습 등)
- 직업/역할 — "어떤 일을 하세요? 직책은?" (도메인 전문성 수준 파악)
- 주요 프로젝트 — "지금 진행 중인 프로젝트가 뭐예요?" (현재 맥락의 중심축)
- 기술 스택 — "어떤 언어, 프레임워크, 도구를 쓰세요?" (구현 맥락)
- 반복되는 작업 패턴 — "가장 자주 하는 작업이 뭐예요?" (핵심 사용 시나리오)

맥락 심화 (중요 — 중간 가중치):
- 작업 환경 — "혼자 작업? 팀? 협업 방식은?" (커뮤니케이션 맥락)
- 품질 기준 — "결과물에서 가장 중요하게 보는 게 뭐예요?" (판단 기준)
- 자주 겪는 문제 — "AI 쓸 때 가장 짜증나는 게 뭐예요?" (보정 우선순위)
- 도메인 제약 — "지켜야 하는 규칙이나 표준이 있어요?" (공공기관 보안 요건 등)
- 선호하는 소통 방식 — "코드 먼저? 설명 먼저? 간결하게? 상세하게?" (응답 스타일)

배경 보충 (참고 — 낮은 가중치):
- 경력 배경 — "이전에 어떤 일을 했어요?" (지식 범위 파악)
- 학습 관심사 — "요즘 배우고 싶은 게 있어요?" (확장 가능한 맥락)
- 개인 선호 — "작업 스타일이나 습관이 있어요?" (개인화 요소)

이 인터뷰는 자연스러운 대화 형식으로 진행되며, Synapse가 답변을 분석하여 자동으로 중요도를 분류한다. 수집된 정보는 맥락 풀에 가중치가 차등 적용되어 저장되므로, 이후 벡터 검색 시 핵심 정보가 우선 연결된다.

인터뷰 대화 자체도 턴별로 벡터화되어 맥락 풀에 추가된다.

문서만으로는 "이 사람이 어떤 맥락에서 이 기술을 쓰는지" 뉘앙스가 부족하고, 대화만으로는 코드나 규칙 같은 구체적 맥락이 부족하다. 둘을 합쳐야 첫 세션부터 의미 있는 연결이 가능하다.

온보딩 데이터의 연결 가중치는 전부 초기값(0)에서 시작하고, 사용하면서 강화/약화가 쌓인다. 온보딩은 학습 데이터의 시작점이다.

### Phase 1: 자동 연결 + 기본 대화 (①②)
- 대화 턴 벡터 임베딩 자동 저장
- 새 질문 시 유사도 기반 관련 턴 자동 선별
- 선별된 턴만으로 messages 조립 → Ollama API 전송
- 전체 주입 모드와 선택적 연결 모드 비교 토글 (효과 체감용)

### Phase 2: 보정 도구 (③)
- 🚩 플래그 + 이유 입력
- 이분 탐색으로 원인 턴 탐지
- 대안 제시 + 턴 교체 + 재생성
- 턴별 포함/제외 토글, 내용 수정
- 스냅샷 저장/불러오기, 브랜치

### Phase 3: 연결 강화/약화 (④)
- 성공/실패 데이터 기반 연결 가중치 업데이트
- 주제별 최적 맥락 조합 자동 추천
- 프롬프트 코치 피드백

## 선행 연구 분석

### MemGPT (2023, UC Berkeley)
- OS의 가상 메모리 개념으로 LLM이 자기 메모리를 계층적으로 관리
- 메인 컨텍스트(RAM) ↔ 외부 컨텍스트(디스크) 간 페이징
- **한계**: AI 자율 관리 — 사용자 컨트롤 없음, 턴 단위 관리 아님, 디버깅 불가

### ContextBranch (2024.12, 학술 논문)
- Git 버전 관리를 LLM 대화에 적용 (checkpoint, branch, switch, inject)
- 브랜치 대화가 포커스 +4.6%, 컨텍스트 인식 +6.8% 향상, 컨텍스트 크기 58.1% 감소
- **한계**: 수동 선택 번거로움, 디버깅/학습 루프 없음
- **주목**: 향후 과제로 "임베딩 유사도 기반 자동 인사이트 식별" 제안 → Synapse의 자동 연결 개념과 일치

### A-Mem (2025)
- 상호 연결된 메모리 네트워크 + 선택적 top-k 검색
- MemGPT 대비 토큰 85~93% 감소, 멀티홉 추론 2배 향상
- **한계**: 사용자 컨트롤 없음, 디버깅 없음

### 업계 실무 (LangChain, LlamaIndex 등)
- 선택적 컨텍스트 주입, 벡터 인덱싱 대화 검색 등 기술적으로는 가능
- **한계**: 전부 개발자용 백엔드 도구, 사용자 대면 UI 없음

### Synapse가 추가하는 빠진 퍼즐 조각

| | MemGPT | ContextBranch | A-Mem | **Synapse** |
|---|---|---|---|---|
| 선택적 연결 | AI 자율 | 사용자 수동 | AI 자동 | **자동 + 사용자 보정** |
| 디버깅 | ❌ | ❌ | ❌ | **✅ 원인 탐지 + 교정** |
| 연결 강화/약화 | ❌ | ❌ | ❌ | **✅ 학습 루프** |
| 사용자 UI | CLI | 프리미티브 | API | **✅ 시각적 에디터** |
| 턴 단위 관리 | ❌ | ✅ | ❌ | **✅ 턴 + 턴 내부 부분 편집** |
| 실행 전 검토 | ❌ | ❌ | ❌ | **✅ 컨텍스트 프리뷰** |
| 의도 정제 | ❌ | ❌ | ❌ | **✅ 듀얼 세션 (A→B)** |
| 진입 장벽 | 높음 | 높음 | 높음 | **낮음 (로컬, 무료)** |

기술적 기반은 선행 연구에서 검증됨. 핵심 난관은 "관련 턴 자동 판별 정확도"이며, Synapse의 "자동 추천 + 사용자 보정 → 학습" 하이브리드가 현실적 해법.

## 핵심 가치 정리

Synapse는 AI를 더 똑똑하게 만드는 것이 아니다. **AI가 원래 가진 지능을 100% 발휘하게 해주는 레이어**다.

똑똑한 사람한테 관련 없는 서류 1000장을 책상 위에 쌓아놓고 "이것도 참고해"라고 하면 멍청해 보인다. 필요한 서류 3장만 딱 놓아주면 바로 핵심을 짚는다. 지금 AI가 멍청해 보이는 순간들은 대부분 모델의 한계가 아니라 맥락의 문제다.

## 자원 요구사항

| 구성 요소 | 필요 자원 | 비고 |
|---|---|---|
| 임베딩 모델 | CPU만으로 충분 | nomic-embed-text 등 경량 모델 |
| 벡터 DB | RAM 수십 MB | ChromaDB (SQLite 기반, 서버 불필요) |
| LLM 추론 | Ollama (CPU/GPU) | 오히려 선택적 연결으로 토큰 감소 → 추론 더 빨라짐 |
| 프론트엔드 | Vite + React | 경량 |
| 저장소 | SQLite | 로컬 파일 |

한 명의 사용자가 로컬에서 구동하는 데 특별한 자원이 필요하지 않다. 진입 장벽이 낮다.
