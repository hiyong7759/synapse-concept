**최종 업데이트**: 2026-04-19

## 1. 시스템 관통 원칙

프로젝트 전체를 관통하는 핵심 원칙. 모든 구현·문서는 이 원칙을 위배할 수 없다.

### 존재론

1. **언급된 것만 존재한다.** 시드 데이터·기본 사전·시스템 노드 없음. 치환 로직은 노드 생성 수단이지 미리 넣어둔 데이터가 아니다. 시냅스 = 사용자의 언급 공간과 정확히 일치.
2. **노드는 원자다.** 인물·장소·수치·상태·행위 전부 노드. 품사 무관. 하나의 개념 = 하나의 노드. 중복 생성 금지. 대소문자, 언어 불문 같은 개념이면 같은 노드.
3. **외부 참조는 저장하지 않는다.** 행정구역 등 표준 공공 데이터는 그래프에 노드로 두지 않고 필요 시점에 외부 API로만 조회. 매칭용 메타데이터도 노드에 부착하지 않는다.

### 구조

4. **엣지는 문장을 뛰어넘는 의미 관계다.** 조사 기반 엣지 폐기 — 문장 내 문법 구조는 원문 `sentences`로 이미 보존된다. 엣지는 `similar / cooccur / cause / contain` 같은 의미 관계만 담으며, **사후 생성 + 사용자 승인**으로만 `edges`에 들어간다.
5. **엣지는 유방향이다.** `source_node_id → target_node_id`. BFS 탐색은 양방향 조회, 저장은 방향 있음.
6. **동의어/다국어는 aliases.** React Native ↔ 리액트 네이티브 → `aliases`로 매칭 강화.
7. **meta 컬럼·JSON 필드를 노드에 추가하지 마라.** "기간", "전공", "담당업무" 같은 상세 정보도 별도 노드 + 역참조로 표현한다. 단 `node_categories`(표시·필터 태깅용)는 예외.

### 저장과 승인

8. **저장과 합성의 분리.** 자동 저장 범위는 원문 sentence + 노드 + 노드↔문장 역참조(`node_mentions`) + 치환 실패 토큰(`unresolved_tokens`)까지. 카테고리·엣지·별칭·병합 등 **모든 의미 합성은 사용자 승인**(`/review`)을 거쳐야 DB에 반영된다. 사용자가 인지하지 못한 채 쌓이는 지식은 사용자의 지식이 아니다.
9. **DB에 있는 것은 전부 승인된 것.** `edges`, `node_categories`, `aliases`는 모두 승인된 결과만 담는다. 승인 대기 중인 제안은 저장하지 않고 `/review` 호출 시 런타임에 도출한다.
10. **입력 단위 = 마크다운 구조화된 게시물.** 세션·맥락 경계 자동 추론 없음. 평문은 LLM이 마크다운 구조화 초안을 제안하고 사용자 확정 후 저장.

### 동작

11. **도메인은 관찰하는 것이다.** 연결이 많아진 노드가 허브가 되고, 허브가 도메인으로 기능한다. 사전 정의하지 않는다.
12. **지능체는 분리되어 있다.** 시냅스 = 저장 + 인출 엔진. 사고·합성은 외부 지능체(외부 고급 LLM, 로컬 경량 LLM, 사용자 본인)가 수행. 시냅스는 LLM 없이도 독립 동작해야 한다.
13. **모든 질문은 개인 맥락이 있으면 더 나은 답을 만든다.**

## 2. 개인 / 조직 모드 원칙

### 모드 사용 원칙 (그래프 레벨)

- **개인 모드: 1인칭은 문장에 있을 때만 "나" 노드.** 문장에 명시된 1인칭(나/내/저/제)만 "나" 노드로 저장. 문장에 없는 1인칭을 추가하는 것은 금지.
- **조직 모드: 주어 항상 명시.** 누가 말했는지가 핵심. 1인칭은 사용자 이름으로 치환. 주어 불명확 시 LLM이 질문.

### 네트워크 원칙 (앱·조직 관계)

- **개인 그래프**: 온디바이스. 완전히 사용자 소유.
- **조직 그래프**: 조직 서버. 소속 시 접근 가능.
- **개인 → 조직**: 직접 접근 가능.
- **조직 → 개인**: 직접 접근 **불가**. 요청-응답 프로토콜을 통해서만 정보를 받는다.
- **다중 소속**: 개인은 여러 조직에 동시에 속할 수 있다.

상세: `docs/DESIGN_GRAPH.md`, `docs/DESIGN_ORG.md`, `docs/DESIGN_APP.md`

## 3. /review 검토 원칙

승인 대기 상태 관리 방식. 상세: `docs/DESIGN_REVIEW.md`

1. **승인 대기 전용 테이블을 만들지 않는다** — 유일한 예외는 `unresolved_tokens` 하나.
2. **제안은 요청 시점에 런타임 도출** — `GET /review` 호출 시 `engine/suggestions.py`가 DB 쿼리 + 필요 시 LLM 호출로 리스트 생성. 결과는 DB에 쓰지 않는다.
3. **승인 즉시 최종 테이블에 반영** — `edges / node_categories / aliases`에는 승인된 것만 존재.
4. **LLM 없이도 동작** — 규칙 기반 섹션은 LLM 의존 없이 쿼리만으로 채워진다.

이 설계는 시스템 관통 원칙 §9("DB에 있는 것은 전부 승인된 것")를 `edges`, `node_categories`, `aliases` 전반에 일관되게 적용한 결과다. `status / origin / kind / action` 같은 중간 분류 컬럼은 모두 불필요하다.

## 4. 카테고리·인접 맵 원칙

인접 맵 구성 방침. 상세: `docs/DESIGN_CATEGORY.md`

1. **소분류 레벨** — 대분류 레벨 연결은 범위가 너무 넓어 노이즈 증가. 소분류로 좁혀 정밀도 확보.
2. **크로스 대분류만** — 같은 대분류 내 소분류 간 연결은 기존 BFS 보완(대분류 전체 조회)으로 이미 처리됨.
3. **단방향 정의, 양방향 적용** — 코드에서 역방향 자동 포함.
4. **약한 연관은 제외** — "질문 시 같이 보면 답이 확실히 나아지는가"가 기준.
5. **데이터 기반 검증** — 데이터가 쌓이면 카테고리 간 엣지 빈도로 보완·조정.

## 5. UI 원칙

채팅·그래프 뷰의 동작 방침. 상세: `docs/DESIGN_UI.md`

1. **그래프 변경 알림은 메시지를 방해하지 않는다** — 말풍선 하단에 조용히 붙는다.
2. **탐색 중(BFS) 상태는 숨기지 않는다** — 진행 중인 노드 탐색을 가볍게 시각화한다.
3. **그래프 뷰는 별도 `/graph` 라우트** — 채팅 상단 바에서 진입.
4. **조직 연결 여부는 항상 상태 인디케이터로 노출.**

## 6. 데이터 정책 원칙

### 날짜 노드 — 한국어 표기로 통일

노드 = 사용자가 일상에서 말하는 단위. ISO(`2026-04-18`)로 입력돼도 본문 sentence와 노드 모두 **한국어 표기**(`2026년`, `4월`, `18일`)로 정규화한다.

- **식별 규칙**: 명시적 키워드(`년/월/일`) 또는 ISO 구분자(`-`)가 있을 때만 날짜로 판단. 단독 4자리 숫자(`2026`)는 오탐 위험으로 노드화하지 않는다.
- **분할 저장**: `2026년 4월 18일` 통째 노드 + `2026년`, `4월`, `18일` 각각 독립 노드 → `node_mentions` 교집합으로 시간 범위 쿼리 지원.

상세 규칙표(정규화 예시): `docs/DESIGN_PIPELINE.md` "날짜 처리" 섹션.
