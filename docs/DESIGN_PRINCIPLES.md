## 1. 시스템 관통 원칙

프로젝트 전체를 관통하는 핵심 원칙. 모든 구현·문서는 이 원칙을 위배할 수 없다.

### 존재론

1. **언급된 것만 존재한다.** 시드 데이터·기본 사전·시스템 노드 없음. 치환 로직은 노드 생성 수단이지 미리 넣어둔 데이터가 아니다. 시냅스 = 사용자의 언급 공간과 정확히 일치.
2. **노드는 원자다.** 인물·장소·수치·상태·행위 전부 노드. 품사 무관. 하나의 개념 = 하나의 노드. 중복 생성 금지. 대소문자, 언어 불문 같은 개념이면 같은 노드.
3. **외부 참조는 저장하지 않는다.** 행정구역 등 표준 공공 데이터는 하이퍼그래프에 노드로 두지 않고 필요 시점에 외부 API로만 조회. 매칭용 메타데이터도 노드에 부착하지 않는다.

### 구조

4. **연결은 카테고리 공유 + 공출현으로 창발한다.** 엣지 테이블 폐기. 노드 간 연결은 ① `node_mentions` 공출현(같은 sentence), ② `node_categories` 공유(같은 분류), ③ 카테고리 인접 맵(한 홉 확장) 세 경로로만 드러난다. 의미 관계(`cause`/`avoid`/`similar` 등)는 `sentences` 원문에 이미 있고, 해석은 외부 지능체(원칙 11) 몫이다. 시냅스의 본질은 해부학적 링크가 아니라 **함께 활성화되는 경향**(Hebbian)이며, 공출현·카테고리가 그 실체다.
5. **동의어/다국어는 aliases.** React Native ↔ 리액트 네이티브 → `aliases`로 매칭 강화.
6. **meta 컬럼·JSON 필드를 노드에 추가하지 마라.** "기간", "전공", "담당업무" 같은 상세 정보도 별도 노드 + 역참조로 표현한다. 단 `node_categories`(표시·필터 태깅용)는 예외.

### 저장과 출처

7. **저장은 자동, 출처는 기록한다.** 원문 sentence + 노드 + `node_mentions` + `node_categories` + `aliases`는 모두 자동 저장한다. 각 레코드는 `origin` 컬럼(`user` / `ai` / `rule`)으로 출처가 식별되며, 사용자는 언제든 AI·규칙 생성물을 수정·삭제할 수 있다. **치환 실패 지시어(`unresolved_tokens`)와 파괴적 작업(노드 병합 등)만** `/review` 승인을 거친다.
8. **DB의 모든 레코드는 출처가 식별된다.** `node_categories`, `aliases`에는 `origin`이 붙는다. UI·`/review`는 이 값으로 필터링해 "AI가 자동으로 단 것만 훑어보기" 같은 검토 뷰를 제공한다. `rule`은 규칙 오류 추적용, `ai`는 LLM 추론 오류 검수용이다.
9. **입력 단위 = 마크다운 구조화된 게시물.** 세션·맥락 경계 자동 추론 없음. 평문은 LLM이 마크다운 구조화 초안을 제안하고 사용자 확정 후 저장.

### 동작

10. **도메인은 관찰하는 것이다.** 공출현·카테고리 연결이 많아진 노드가 허브가 되고, 허브가 도메인으로 기능한다. 사전 정의하지 않는다.
11. **지능체는 분리되어 있다.** 시냅스 = 저장 + 인출 엔진. 사고·합성(인과·유사·회피 같은 의미 해석 포함)은 외부 지능체(외부 고급 LLM, 로컬 경량 LLM, 사용자 본인)가 수행. 시냅스는 LLM 없이도 독립 동작해야 한다.
12. **모든 질문은 개인 맥락이 있으면 더 나은 답을 만든다.**

## 2. 개인 / 조직 모드 원칙

### 모드 사용 원칙 (하이퍼그래프 레벨)

- **개인 모드: 1인칭은 문장에 있을 때만 "나" 노드.** 문장에 명시된 1인칭(나/내/저/제)만 "나" 노드로 저장. 문장에 없는 1인칭을 추가하는 것은 금지.
- **조직 모드: 주어 항상 명시.** 누가 말했는지가 핵심. 1인칭은 사용자 이름으로 치환. 주어 불명확 시 LLM이 질문.

### 네트워크 원칙 (앱·조직 관계)

- **개인 하이퍼그래프**: 온디바이스. 완전히 사용자 소유.
- **조직 하이퍼그래프**: 조직 서버. 소속 시 접근 가능.
- **개인 → 조직**: 직접 접근 가능.
- **조직 → 개인**: 직접 접근 **불가**. 요청-응답 프로토콜을 통해서만 정보를 받는다.
- **다중 소속**: 개인은 여러 조직에 동시에 속할 수 있다.

상세: `docs/DESIGN_HYPERGRAPH.md`, `docs/DESIGN_ORG.md`, `docs/DESIGN_APP.md`

## 3. /review 검토 원칙

자동 저장 + 출처 추적 구조에서 `/review`는 **"승인 대기 해소"** + **"AI·규칙 생성물 검토"** 두 역할을 한다.

상세: `docs/DESIGN_REVIEW.md`

1. **자동 저장 + 출처 추적** — 카테고리·별칭은 자동으로 DB에 들어가며, `origin` 컬럼으로 `user` / `ai` / `rule`을 구분한다.
2. **`unresolved_tokens`만 승인 대기** — 치환 실패 지시어는 저장 시점 일회성 이벤트라 런타임 재생성 불가. 유일한 "저장은 안 되고 승인 후 편입"되는 예외.
3. **파괴적 작업만 승인 유지** — 노드 병합(`merge_nodes`) 같은 되돌릴 수 없는 작업은 `/review` 승인 흐름 유지.
4. **검토 뷰 — AI·규칙 생성물 목록** — `/review`는 `origin='ai'` / `'rule'` 필터 뷰를 제공. 사용자는 항목 단위로 즉시 수정·삭제. 대기 테이블 없음 — 이미 DB에 있으므로 조회만.
5. **LLM 없이도 동작** — 규칙 기반 섹션은 LLM 의존 없이 쿼리만으로 채워진다.

이 설계는 v13("사용자 승인만 DB 반영") → v14("자동 저장 + 출처 추적") → v15("엣지 테이블 폐기, 연결은 카테고리·공출현으로 창발")로 전환된 결과다. 잘못된 자동 저장은 사용자가 목록 뷰에서 "이건 삭제"로 즉시 바로잡을 수 있고, `rule` 태그는 규칙 자체의 오류 추적에도 활용된다.

## 4. 카테고리·인접 맵 원칙

인접 맵 구성 방침. 상세: `docs/DESIGN_CATEGORY.md`

1. **소분류 레벨** — 대분류 레벨 연결은 범위가 너무 넓어 노이즈 증가. 소분류로 좁혀 정밀도 확보.
2. **크로스 대분류만** — 같은 대분류 내 소분류 간 연결은 기존 BFS 보완(대분류 전체 조회)으로 이미 처리됨.
3. **단방향 정의, 양방향 적용** — 코드에서 역방향 자동 포함.
4. **약한 연관은 제외** — "질문 시 같이 보면 답이 확실히 나아지는가"가 기준.
5. **데이터 기반 검증** — 데이터가 쌓이면 카테고리 간 **문장 바구니 공출현 빈도**로 보완·조정.

## 5. UI 원칙

채팅·하이퍼그래프 뷰의 동작 방침. 상세: `docs/DESIGN_UI.md`

1. **하이퍼그래프 변경 알림은 메시지를 방해하지 않는다** — 말풍선 하단에 조용히 붙는다.
2. **탐색 중(BFS) 상태는 숨기지 않는다** — 진행 중인 노드 탐색을 가볍게 시각화한다.
3. **하이퍼그래프 뷰는 별도 `/hypergraph` 라우트** — 채팅 상단 바에서 진입.
4. **조직 연결 여부는 항상 상태 인디케이터로 노출.**

## 6. 데이터 정책 원칙

### 날짜 노드 — 한국어 표기로 통일

노드 = 사용자가 일상에서 말하는 단위. ISO(`2026-04-18`)로 입력돼도 본문 sentence와 노드 모두 **한국어 표기**(`2026년`, `4월`, `18일`)로 정규화한다.

- **식별 규칙**: 명시적 키워드(`년/월/일`) 또는 ISO 구분자(`-`)가 있을 때만 날짜로 판단. 단독 4자리 숫자(`2026`)는 오탐 위험으로 노드화하지 않는다.
- **분할 저장**: `2026년 4월 18일` 통째 노드 + `2026년`, `4월`, `18일` 각각 독립 노드 → `node_mentions` 교집합으로 시간 범위 쿼리 지원.

상세 규칙표(정규화 예시): `docs/DESIGN_PIPELINE.md` "날짜 처리" 섹션.

## 7. 파인튜닝 작업 원칙

학습·파인튜닝 작업 시 지켜야 할 원칙. 체크리스트·하이퍼파라미터·태스크 정의는 `docs/DESIGN_FINETUNE.md` 참고.

1. **추측하지 마라. 검색해라. 실행 전에 검증해라.** 일반적 ML 지식으로 추측하지 않는다. 이 프로젝트엔 확정된 설계·하이퍼파라미터·실패 기록·성공 패턴이 이미 문서화되어 있다.
2. **MLX 우선, RunPod 폴백.** 학습은 기본적으로 Mac M4 + MLX LoRA로 진행한다. RunPod은 **부득이한 경우에만**(대용량 모델, 장시간 학습, MLX 비호환 연산, 대규모 병렬 실험) 사용한다.
3. **환경은 매번 검증한다.** 문서의 설정이 현재 환경에서 유효한지 실행 전 매번 확인 — 라이브러리 버전, 모델 접근, 데이터 건수, VRAM/메모리, 디스크, 기존 어댑터 백업.
4. **새 환경·버전·데이터는 100 iters 소규모 검증 필수.** 전체 학습 전에 train_loss 감소, eval_loss 트렌드, 출력 포맷을 검증한다. 문제 있으면 원인 분석 우선.
5. **하이퍼파라미터를 임의로 정하지 않는다.** lr·layers·alpha·batch 등은 이미 확정된 값이 있다. 이전 실패 접근(전체 레이어 LoRA, alpha=160 등)도 반복하지 않는다.
6. **"설정 완료"는 동작 확인 후에만 말한다.** 실제 1회 실행으로 검증 후 보고한다. 에러 발생 시 로그 일부만 보고 판단하지 않고 성공/실패 케이스를 비교한다.
