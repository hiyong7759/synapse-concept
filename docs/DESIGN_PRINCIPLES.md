## 1. 시스템 관통 원칙

### 존재론

1. **언급된 것만 존재한다.** 시드 데이터·기본 사전·시스템 노드 없음. 치환 로직은 노드 생성 수단이지 미리 넣어둔 데이터가 아니다. 시냅스 = 사용자의 언급 공간과 정확히 일치.
2. **노드는 원자다.** 인물·장소·수치·상태·행위 전부 노드. 품사 무관. 하나의 개념 = 하나의 노드. 중복 생성 금지. 대소문자, 언어 불문 같은 개념이면 같은 노드. **한국어 용언은 형태소 분석기(Kiwi)의 어간(lemma)으로 정규화**한다 — `아파서` / `아프` / `아팠어` 는 같은 노드 `아프`. **외래어·영문·복합 고유명사는 Kiwi 형태소 분석 결과(쪼갠 조각)를 그대로 저장**한다 — `React Native` 는 `React`, `Native` 두 노드로 들어가며, 같은 sentence 에 공출현하므로 원칙 4(문장 바구니) 로 연결이 창발한다. 다국어·표기 변형은 `aliases` 로 연결(원칙 5).
3. **외부 참조는 저장하지 않는다.** 행정구역 등 표준 공공 데이터는 하이퍼그래프에 노드로 두지 않고 필요 시점에 외부 API로만 조회. 매칭용 메타데이터도 노드에 부착하지 않는다.

### 구조

4. **연결은 카테고리 공유 + 공출현으로 창발한다.** 엣지 테이블 없음. 노드 간 연결은 ① `node_sentence_mentions` 공출현(같은 sentence), ② `node_category_mentions`(노드 ↔ 카테고리 단일 매핑), ③ 사용자 heading 계층(`categories` adjacency list + `sentence_categories` 문장 매핑, 재귀 CTE 서브트리) — 세 경로로만 드러난다. 의미 관계(`cause`/`avoid`/`similar` 등)는 `sentences` 원문에 이미 있고, 해석은 외부 지능체(원칙 11) 몫이다. 시냅스의 본질은 해부학적 링크가 아니라 **함께 활성화되는 경향**(Hebbian)이며, 공출현·카테고리가 그 실체다.
5. **동의어/다국어는 aliases.** React Native ↔ 리액트 네이티브 → `aliases`로 매칭 강화.
6. **meta 컬럼·JSON 필드를 노드에 추가하지 마라.** "기간", "전공", "담당업무" 같은 상세 정보도 별도 노드 + 역참조로 표현한다. 단 `node_category_mentions`(노드 ↔ 카테고리 멤버십, 표시·필터 태깅용)는 예외.

### 저장과 출처

7. **저장은 자동, 출처는 기록한다 — 저장은 순수 기록.** 원문 sentence + 노드 + `node_sentence_mentions` + `sentence_categories` + `node_category_mentions` + `aliases`는 모두 자동 저장한다(`categories` 마스터는 origin 없음 — 삭제 경로 부재). 각 레코드는 `origin` 컬럼(`user` / `ai` / `system` / `external`)으로 출처가 식별되며, 사용자는 언제든 AI·시스템·외부 생성물을 수정·삭제할 수 있다. **치환 실패 지시어(`unresolved_tokens`)와 파괴적 작업(노드 병합, 통찰 삭제 등)만** `/review` 승인을 거친다. 저장 시점 해석·판정은 전면 배제 — 시점 해석은 인출 시점 LLM 책임.
8. **DB의 모든 레코드는 출처가 식별된다 — 파괴적 자동 작업 없음.** `sentence_categories`, `node_category_mentions`, `aliases`, `sentences` 에는 `origin`이 붙는다. UI·`/review`는 이 값으로 필터링해 "AI가 자동으로 단 것만 훑어보기" 같은 검토 뷰를 제공한다. 출처별 역할:
    - `user` — 사용자가 직접 명시 (마크다운 heading, 수동 등록)
    - `ai` — LLM 추론 (카테고리 분류 워커)
    - `system` — 결정론적 엔진 규칙 (Kiwi lemma 정규화, 날짜 정규화, 부정부사 감지, 인칭대명사 별칭 시드)
    - `external` — 외부 API (Wikidata altLabel)
    - `insight` — 사용자 승격 통찰 (`sentences.origin` 전용 — 본체 sentence 식별, 편집 불가)

   자동 deactivate·pending 같은 파괴적 자동 판정 없음. 무효화는 `update_sentence`·`delete_sentence` 로 사용자가 직접.

9. **post = 세션 그릇. 입력은 노트 단일, 의미 처리는 사용자 명시.** 한 개의 `posts` 행은 **사용자 활동의 세션 그릇**이며, 그 안에 `sentences`가 누적된다. 사용자 메시지·비서 응답·질문·답 모두 같은 post 에 섞여 쌓이고, 화자는 `sentences.role` 로 구분된다. post 는 사용자가 **명시적으로 나갈 때까지 열려있고**, 목록(번호·제목·수정시간)에서 언제든 재진입·이어서 쓰기·편집이 가능하다. 시간 기반 자동 경계 없음.

   세션 그릇은 세 종류이며 사용자 액션으로 결정된다 (자동 판정 없음):
   - **`note`** — `/note` 화면에서 사용자가 입력하면 자동 생성되는 축적 그릇. 한 줄 메모도 긴 마크다운도 같은 그릇. 사용자는 입력 형식을 인지·선택하지 않으며, heading·list 같은 마크다운 구조를 쳐도 평문을 쳐도 동일하게 받는다. save-pronoun·메타 필터·Kiwi 노드 편입은 단일 경로로 일관 적용된다.
   - **`synapse`** — 축적된 지식 사이 새 연결을 발화시키는 인출·융합 세션. 질문·답이 역사로 기록되며 기본적으로 노드·바구니 편입 없음.
   - **`insight`** — 사용자가 시냅스 세션의 메시지를 **명시적으로 "통찰"로 승격**시켜 생성되는 고유 그릇. `posts` 1 행 = 통찰 1 개 = 본체 `sentence` 1 개 (원칙 15 참조). 본체는 편집 불가, 허브로 자동 연결.

   **저장은 두 층으로 분리된다:**
   - **자동저장** — 입력 1.5초 디바운스 + 페이지 이탈 시 백그라운드. `posts.source` 만 갱신. **LLM 호출 없음**. 손실 방지 목적. 모바일 배터리·데이터 부담 거의 없음.
   - **의미 처리** — 사용자 명시 트리거(⌘S / "정리" 버튼 / 재진입 시 제안). `sentences` 재계산 + Kiwi 노드 추출 + LLM 정정 후보 생성. 원본은 항상 보존되고, LLM 정정은 인라인 카드로 제안만 — 사용자 [적용] 클릭이 있어야 source/sentences 가 갱신된다 (자동 적용 금지).

   이 분리의 근거는 원칙 13(로컬 모델 제약) — 매 입력마다 LLM 호출은 모바일에서 지속 불가. 사용자가 "이 메모를 의미 있게 닫는다" 신호를 줄 때만 의미 처리 비용을 치른다.

   내용 기반 자동 판정 없음 — 평문을 마크다운으로 강제 변환 금지, `has_heading()` 자동 분기 금지. 입력 형식 차이는 LLM 의미 처리 단계가 흡수한다. 파이프라인 세부는 `docs/DESIGN_INPUT_MODES_AND_RETRIEVAL.md` · `docs/DESIGN_PIPELINE.md` 참조.

### 동작

10. **도메인은 관찰하는 것이다 — 해석은 관찰 이후.** 공출현·카테고리 연결이 많아진 노드가 허브가 되고, 허브가 도메인으로 기능한다. 사전 정의하지 않는다.
11. **LLM 은 핵심이지만 호출은 의존성 주입으로 분리한다.** 시냅스의 검색 필터·답변 합성·키워드 확장은 LLM 이 수행한다. 다만 엔진의 그래프 코어 (노드·sentence·카테고리·BFS 골격) 는 LLM 모듈을 직접 import 하지 않는다 — LLM 이 필요한 지점에선 **콜백 함수를 외부에서 주입** 받아 호출한다. 그래서 ① 콜백을 안 넘기면 LLM 없이도 그래프 코어가 컴파일·실행 가능 (저장 파이프라인 + Kiwi 노드 추출 + BFS 무필터 모드), ② 갑질 같은 재사용 앱은 자기 도메인 LLM 함수를 주입해 같은 코어를 재활용, ③ 시냅스 본체는 `LlmTasks` 콜백을 주입해 풀 기능 가동. **저장 기본 경로는 Kiwi 단독** — 노드 추출은 Kiwi 형태소 분석만으로 충분하며 LLM 은 `meta-filter`(메타 대화 제거)·`typo-normalize`(정정 후보 생성) 보조 태스크에만 선택적으로 쓴다. 로컬 LLM 사용 불가 시 보조 태스크 전부 skip 하고 Kiwi 경로만으로 저장이 완결된다. **형태소 분석기는 플랫폼별 바인딩으로 분화**한다 — 서버(조직/API)는 `kiwipiepy`(C++ 네이티브), 모바일·데스크톱·웹(개인 앱)은 `kiwi-nlp`(WASM). 토큰 경계·품사 태그·lemma 결과 스키마는 양쪽 동일하게 유지해 파이프라인 로직을 공유한다.
12. **모든 질문은 개인 맥락이 있으면 더 나은 답을 만든다.**

### 로컬 제약과 UI 의 의무

13. **로컬 모델 제약을 인정한다 — 자동 판정보다 사용자 명시.** 시냅스는 정보 주권을 지키기 위해 로컬 모델로 돌아간다. 로컬 모델은 클라우드 대비 품질 편차가 있으므로 "LLM 이 알아서 판정" 으로 풀리지 않는다. 모호한 결정(예: 저장 모드·경계·통찰 승격)은 **사용자 명시**로 결정되어야 하며, 시스템은 자동 판정으로 사용자의 기록을 오염시키지 않는다. 원칙 9의 "사용자가 명시한다" 가 이 제약에서 온 결과다.

14. **UI 의 핵심 의무는 규칙을 쉽게 지키게 돕는 것이다.** 시냅스는 (특히 마크다운 구조화에서) 엄격한 입력 규칙을 요구한다. 규칙을 없애는 게 아니라 **규칙을 지키기 쉽게 만드는 것**이 UI 의 설계 우선순위. 구체적으로:
    ① **자동저장 상태 상시 표시** (변경됨 / 저장 중 / ✓ 저장됨) — 손실 걱정 제거
    ② **의미 처리 트리거 명확** (⌘S / "정리" 버튼) — "지금 LLM 이 정리하고 있다" 가 사용자에게 보여야 함
    ③ **LLM 정정 카드는 제안만, 자동 적용 금지** — 사용자 원문의 정감·의도가 임의로 표준화되지 않도록 [적용]/[무시] 클릭 강제
    ④ **별칭(`aliases`) 등록된 토큰은 정정 후보에서 제외** — 사용자가 의도한 약어·애칭(`스벅` 등) 보존
    ⑤ 규칙 위반(콜론 하나, 빈 value 등)은 편집 중 **실시간 밑줄+설명** (의미 처리 시점에 반영)

    규칙이 필요한 이유(원칙 11 지능체 분리 · 원칙 13 로컬 제약) 를 UI 가 "불편"이 아닌 "자연스러운 흐름"으로 전환해내야 한다.

### 시냅스

15. **시냅스는 축적된 지식 사이에 새 연결이 발화하는 장이며, 그 산물은 허브가 되어 다음 발화의 중력이 된다.** 채팅·마크다운으로 쌓인 문장·노드·바구니가 뉴런 재료라면, `synapse` 모드는 그것들이 함께 점화해 **사용자가 혼자서는 연결시키지 못했던 통찰**을 만들어내는 순간의 UI. LLM 은 축적된 재료에서 관련 있는 것을 당겨와 재조합하는 **촉매**이지 지식의 원천이 아니다. 답은 원문 나열이 아니라 합성물이며, 사용자의 후속 발화로 계속 정제·심화된다.

15-1. **기본 — 역사만 기록.** `synapse` 세션의 질문·답은 post 에 역사로 저장되지만, 자동으로 노드·바구니에 편입되지 않는다. 질문은 사실 언급이 아니기 때문이다 (원칙 1 참고).

15-2. **승격 — 산물이 허브가 된다.** 사용자가 시냅스 세션의 특정 메시지를 "통찰"로 승격시키면:
- 새 `posts` 행(`kind='insight'`, 1 post = 1 본체 sentence)으로 고유 그릇 생성
- 본체 sentence 는 `origin='insight'` 로 표식되고 **본문 편집 불가** (수정은 생성 시점의 의미를 변질시킴)
- 해당 시냅스 세션에서 retrieve 로 당겨진 **모든 노드**와 `node_sentence_mentions`가 자동으로 일괄 연결된다 — 본문 언급 여부 무관. "함께 발화한" 것이 전부 엮이는 Hebbian 구현
- 이후 retrieve 에서 이 통찰과 연결 노드는 **우선 점화**되어 다음 시냅스의 중력이 된다
- 통찰 본문에는 덧붙일 수 없으나, 다른 post 가 같은 노드·카테고리를 공유하면 원칙 4의 창발 규칙으로 자연스럽게 연결된다 (통찰을 참조하는 방향)
- 삭제는 `/review` 승인 경로 (파괴적 작업이므로 원칙 8)

## 2. 개인 / 조직 모드 원칙

### 모드 사용 원칙 (하이퍼그래프 레벨)

- **개인 모드: 1인칭은 문장에 있을 때만 "나" 노드.** 문장에 명시된 1인칭(나/내/저/제)만 "나" 노드로 저장. 문장에 없는 1인칭을 추가하는 것은 금지.
- **조직 모드: 주어 항상 명시.** 누가 말했는지가 핵심. 1인칭은 사용자 이름으로 치환. 주어 불명확 시 LLM이 질문.

### 네트워크 원칙 (앱·조직 관계)

- **개인 하이퍼그래프**: 온디바이스. 완전히 사용자 소유.
- **조직 하이퍼그래프**: 조직 서버. 소속 시 접근 가능.
- **개인 → 조직**: 직접 접근 가능.
- **조직 → 개인**: 직접 접근 **불가**. 요청-응답 프로토콜을 통해서만 정보를 받는다.
- **다중 소속**: 개인은 여러 조직에 동시에 속할 수 있다.

상세: `docs/DESIGN_HYPERGRAPH.md`, `docs/DESIGN_ORG.md`, `docs/DESIGN_APP.md`

## 3. /review 검토 원칙

자동 저장 + 출처 추적 구조에서 `/review`는 **"승인 대기 해소"** + **"AI·규칙 생성물 검토"** 두 역할을 한다.

상세: `docs/DESIGN_REVIEW.md`

1. **자동 저장 + 출처 추적** — 카테고리·별칭은 자동으로 DB에 들어가며, `origin` 컬럼으로 `user` / `ai` / `system` / `external` 을 구분한다.
2. **`unresolved_tokens`만 승인 대기** — 치환 실패 지시어는 저장 시점 일회성 이벤트라 런타임 재생성 불가. 유일한 "저장은 안 되고 승인 후 편입"되는 예외.
3. **파괴적 작업만 승인 유지** — 노드 병합(`merge_nodes`)·통찰 삭제 같은 되돌릴 수 없는 작업은 `/review` 승인 흐름 유지.
4. **검토 뷰 — AI·시스템 생성물 목록** — `/review`는 `origin='ai'` / `'system'` 필터 뷰를 제공. 사용자는 항목 단위로 즉시 수정·삭제. 대기 테이블 없음 — 이미 DB에 있으므로 조회만.
5. **LLM 없이도 동작** — 규칙 기반 섹션은 LLM 의존 없이 쿼리만으로 채워진다.

## 4. 카테고리·인접 맵 원칙

`categories` 마스터가 19 대분류 시드 루트 + 사용자 heading 계층을 동시에 담고, `node_category_mentions` 단일 매핑이 노드 ↔ 카테고리 멤버십을 처리. `sentence_categories` 는 사용자 heading 계층 문장 매핑. 인접 맵은 **19 대분류 시드 카테고리 간 관계 전용** — 사용자 heading 계층은 `categories.parent_id` 재귀 CTE 로 서브트리 확장이 구조적으로 해결되어 인접 맵이 불필요. 상세: `docs/DESIGN_CATEGORY.md`

1. **소분류 레벨** — 대분류 레벨 연결은 범위가 너무 넓어 노이즈 증가. 소분류로 좁혀 정밀도 확보.
2. **크로스 대분류만** — 같은 대분류 내 소분류 간 연결은 기존 BFS 보완(대분류 전체 조회)으로 이미 처리됨.
3. **단방향 정의, 양방향 적용** — 코드에서 역방향 자동 포함.
4. **약한 연관은 제외** — "질문 시 같이 보면 답이 확실히 나아지는가"가 기준.
5. **데이터 기반 검증** — 데이터가 쌓이면 카테고리 간 **문장 바구니 공출현 빈도**로 보완·조정.

## 5. UI 원칙

상세: `docs/DESIGN_UI.md`

1. **자동저장 상태 상시 표시** — `/note` 상단에 `변경됨 / 저장 중… / ✓ 저장됨 (방금)` 같은 짧은 상태 줄. 사용자가 손실 걱정 없이 글에 집중할 수 있게 함.
2. **의미 처리 트리거 명확** — ⌘S 또는 "정리" 버튼이 LLM 정리·정정 후보 생성을 시작한다는 걸 사용자에게 분명히 보여줌. 자동저장과 시각적으로 구분 (예: `⚙ 정리 중…` → `✦ 정리됨`).
3. **LLM 정정 카드는 제안만** — 의미 처리 직후 / 재진입 시 인라인 카드로 노출. 사용자 [적용] 클릭 전엔 원문 그대로 보존. 자동 적용 금지 (원칙 14-③). 별칭 등록 토큰은 카드에 안 뜸 (사용자 정감 보존).
4. **탐색 중(BFS) 상태는 숨기지 않는다** — `/synapse` 의 진행 중인 노드 탐색을 가볍게 시각화.
5. **통찰은 시각적으로 허브임을 드러낸다** — `origin='insight'` sentence 는 앰버 보더·✦ 아이콘으로 구분, `/hypergraph` 뷰에서 허브 크기 강조. 편집 불가 상태를 시각적으로 명시.
6. **3 라우트 단순화** — `/note` (축적 + 해당 노트 그래프 패널), `/synapse` (인출·융합 + 해당 세션 그래프 패널), `/hypergraph` (전체 누적 그래프, 조회 전용). `/review` (검수 전용) 는 후속 PLAN.
7. **조직 연결 여부는 항상 상태 인디케이터로 노출.**

## 6. 데이터 정책 원칙

### 날짜 노드 — 한국어 표기로 통일

노드 = 사용자가 일상에서 말하는 단위. ISO(`2026-04-18`)로 입력돼도 본문 sentence와 노드 모두 **한국어 표기**(`2026년`, `4월`, `18일`)로 정규화한다.

- **식별 규칙**: 명시적 키워드(`년/월/일`) 또는 ISO 구분자(`-`)가 있을 때만 날짜로 판단. 단독 4자리 숫자(`2026`)는 오탐 위험으로 노드화하지 않는다.
- **분할 저장**: `2026년 4월 18일` 통째 노드 + `2026년`, `4월`, `18일` 각각 독립 노드 → `node_sentence_mentions` 교집합으로 시간 범위 쿼리 지원.

상세 규칙표(정규화 예시): `docs/DESIGN_PIPELINE.md` "날짜 처리" 섹션.
