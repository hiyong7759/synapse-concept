## 1. 모드 선택 없음 — `/note` 단일 그릇

v22 2차안에서 입력 모드 분리 (chat / markdown) 는 폐지되었습니다. 모든 입력은 `/note` 라우트의 단일 입력 영역으로 받습니다.

| 입력 성격 | 어떻게 쓰면 되는지 |
|---|---|
| 한 줄·몇 줄 즉흥 메모, 감정 기록 | 그냥 평문으로 적기 |
| 제목·항목·표가 있는 구조화된 글 | §2 4 가지 요소 활용 |
| 공문서·법령·규정·매뉴얼 원문 | §2·§4 템플릿 활용 |
| 레시피·체크리스트·업무 일지 | §4 템플릿 활용 |
| 회의록·학습 정리 | §4 템플릿 활용 |

**핵심**: 평문이든 마크다운이든 같은 그릇 (`note` post) 으로 받습니다. heading 이 있으면 의미 처리 단계가 카테고리로 등록하고, 없으면 자유 문장으로 처리합니다 — 사용자가 모드를 선택하지 않습니다.

## 2. 4가지 요소 판단표 ⭐

UI 툴팁 슬롯 → 작성 중 상시 참조 패널. 이 표 하나만 보면 됩니다.

| 내용 성격 | 쓰는 법 | 예시 | 저장 결과 |
|---|---|---|---|
| 분류·주제 제목 (여러 줄) | `# 제목`, `## 소제목` | `## 제4장 인사` | **카테고리 경로** 등록 (문장 저장 안 됨) |
| 분류 경로 한 줄 입력 | `# 분류1.분류2.분류3` | `# 취업규칙.제1장 총칙.제1조` | 여러 단계 **한번에** 등록 |
| 용어 = 정의 | `- key:: value` | `- 보직:: 직원의 자질에 따라...` | 용어·정의가 **묶여서** 한 문장으로 |
| 속성 = 값 | `- key:: value` | `- 개정일:: 2025-04-30` | 메타데이터 문장 |
| 병렬 나열 (2개 이상) | `- value` | `- 이력서 1통`<br>`- 자기소개서 1통` | 각 줄이 **독립 문장** |
| 단일 문장·문단 | 그냥 쓰기 | `이 규칙은 ... 목적으로 한다.` | 문단 단위 문장 |

**핵심 포인트 4개**
- `::` **두 개**여야 key-value로 인식 — `:` 하나는 일반 문장으로 떨어짐
- 항목이 **1개뿐**이면 `- ` 없이 그냥 문장
- 조항·섹션 제목처럼 "그 아래 문장들이 소속되는" 표시는 **반드시 heading**(`##`, `###`)으로
- 한 줄 경로에서 **`.` 은 단계 구분자** — 분류 이름에 `.` 은 넣지 않기 (`Ver 1.0`, `2026.04.23` 같은 점 포함 이름은 본문에만 사용)

## 3. 언제 `#`(heading)으로 올릴까

UI 경고 슬롯 → "이 제목을 heading으로 올릴까요?" 제안 트리거

**기준** — 이 제목 **아래 여러 줄**이 같은 분류에 속하는가?

- ✅ heading: 장·절·조, 챕터, 주제 단위
- ❌ 자유 문장: 한 번만 등장하는 강조, 단일 문장 전환

**단계 (`#` 개수만큼 하위 단계 — 필요한 만큼 자유롭게)**

| 단계 | 용도 | 예시 |
|---|---|---|
| `#` | 문서 전체 주제 | `# 취업규칙` |
| `##` | 장·큰 묶음 | `## 제1장 총칙` |
| `###` | 절 | `### 제1절 인사위원회` |
| `####` | 조항 | `#### 제13조 (구성)` |
| `#####` 이상 | 더 세분화가 필요할 때 | — |

### 한 줄 경로 입력 (점 구분자)

여러 줄로 쓰는 대신 **`.` 로 단계 구분**해서 한 줄에 경로를 지정할 수 있습니다. 빠른 메모·회의록에 편리합니다.

```markdown
# 취업규칙.제1장 총칙.제1조 (목적)
이 규칙은 ... 목적으로 한다.
```

→ 저장되는 경로는 `취업규칙 > 제1장 총칙 > 제1조 (목적)` 3단계로 자동 분해.

**동작 규칙**

- `.` 만나면 경로가 새로 세팅됨 (이전 heading 맥락 리셋)
- `#` 개수는 **무시**됨 — `# A.B.C` 와 `### A.B.C` 결과 동일
- 뒤에 이어지는 `##` 등은 점 경로에 맞춰 덮어쓰기
  ```markdown
  # A.B.C     ← 경로: A > B > C
  ## D        ← 경로: A > D  (C 자리가 D로 바뀜, B 이하 pop)
  ```

**`.` 은 단계 구분자 — 분류 이름에 `.` 넣지 않기**

```markdown
❌  # Ver 1.0       → "Ver 1" > "0" 2단계로 쪼개짐 (원치 않는 해석)
✅  # Ver 1-0        → 단일 분류 "Ver 1-0"
✅  ## Ver 1.0       → 여러 줄 heading이면 점이 있어도 한 단계로 취급
```

점이 포함된 이름(버전·날짜)을 분류로 쓰고 싶으면 **여러 줄 heading**을 쓰거나 **하이픈(`-`)·언더스코어(`_`)** 로 대체하세요.

## 4. 문서 유형별 템플릿

UI 템플릿 삽입 버튼 슬롯 → "법령 / 회의록 / 레시피 / 사양서 / 일지" 드롭다운

### 4.1 법령·규정·정책

```markdown
# 취업규칙

- 회사:: 주식회사 더나은
- 개정일:: 2025-04-30
- 시행일:: 2022-04-01

## 제1장 총칙

### 제1조 (목적)

이 규칙은 ... 을 정함을 목적으로 한다.

### 제2조 (적용범위)

- 이 규칙은 ... 에 적용한다.
- 다만, ... 의 경우에는 제외한다.
```

### 4.2 회의록

```markdown
# 2026-04-23 카테고리 재설계 회의

- 일시:: 2026-04-23 14:00
- 참석자:: 김지원, 박진호, 이수민
- 안건:: v20 스키마 전환

## 논의 내용

### 1안 — 카테고리 테이블 분리

현재 `node_categories`는 ... 한계가 있다. 테이블을 분리하면 ...

### 2안 — JSON 필드 유지

...

## 결정 사항

- 1안 채택
- 박진호가 M1 구현, 다음 주까지 완료
```

### 4.3 레시피

```markdown
# 김치찌개

- 조리시간:: 30분
- 인분:: 2인분
- 난이도:: 쉬움

## 재료

- 김치:: 200g
- 돼지고기:: 150g
- 두부:: 반 모
- 대파:: 1대

## 조리 순서

- 팬에 기름을 두르고 돼지고기를 볶는다.
- 김치를 넣고 5분간 볶는다.
- 물 500ml를 붓고 끓인다.
- 두부와 대파를 넣고 10분 더 끓인다.
```

### 4.4 제품 사양서

```markdown
# iPhone 16 Pro

- 출시일:: 2024-09-20
- 제조사:: Apple
- 가격:: 1,550,000원

## 디스플레이

- 크기:: 6.3인치

- 재질:: Super Retina XDR OLED

## 성능

칩셋은 A18 Pro. 이전 세대 대비 15% 성능 향상. 8코어 CPU · 6코어 GPU.
```

### 4.5 일지·업무 기록

```markdown
# 2026-04-23 업무 일지

- 날씨:: 맑음
- 기분:: 보통

- PLAN-004 M4 리뷰
```

## 5. 자주 하는 실수 ⚠️

UI 실시간 린터 슬롯 → 빨간 밑줄 + 마우스오버 툴팁

### 실수 1 — 콜론을 하나만 씀

```markdown
❌  - 보직: 직원의 자질에 따라 ...     일반 리스트로 강등 (파서 매칭 실패)
✅  - 보직:: 직원의 자질에 따라 ...    용어-정의로 저장
```

### 실수 2 — 조항·섹션 제목을 자유 문장으로 흘려둠

```markdown
❌  ## 제1장 총칙
    제1조 (목적)
    이 규칙은 ...

✅  ## 제1장 총칙
    ### 제1조 (목적)
    이 규칙은 ...
```
이유 — 자유 문장으로 두면 "제1조" 분류가 생기지 않아 조항 경계가 뭉개집니다.

### 실수 3 — 같은 줄에 제목과 본문 섞기

```markdown
❌  ## 제1조 (목적) 이 규칙은 ... 목적으로 한다.

✅  ## 제1조 (목적)
    이 규칙은 ... 목적으로 한다.
```

### 실수 4 — 분류 이름에 `.` 넣기

```markdown
❌  # Ver 1.0        → "Ver 1" 과 "0" 두 단계로 쪼개져 저장됨
✅  # Ver 1-0         → 단일 분류 "Ver 1-0"
✅  ## Ver 1.0        → 여러 줄 heading이면 점 있어도 한 단계로 취급
```
이유 — 한 줄 경로에서 `.` 은 단계 구분자로 해석됩니다. 버전·날짜 등 점 포함 이름을 분류로 쓰려면 하이픈(`-`)·언더스코어(`_`)로 바꾸거나 여러 줄 heading을 쓰세요.

### 실수 5 — 연속 점 `..` 또는 시작·끝 점

```markdown
❌  # A..B            → "A" > "" > "B" 중간에 빈 분류 생성
❌  # .A.B            → 첫 단계가 빈 이름
❌  # A.B.            → 마지막 단계가 빈 이름
✅  # A.B             → 깔끔하게 두 단계
```
이유 — 빈 분류는 인출·검색에서 혼란을 일으킵니다. 점은 단계 사이에 **한 번씩만**.

### 실수 6 — `- key::` 뒤 값이 비어있음

```markdown
❌  - 개정일::         → key-value 인식 실패, 일반 리스트로 강등
✅  - 개정일:: 미정     → 값이 있어야 key-value로 저장됨
✅  - 개정일:: (미정)   → 괄호로 "아직 없음" 명시
```
이유 — key-value 파서는 `::` 앞뒤에 **둘 다** 내용이 있어야 매칭합니다. 비워두려면 "미정"·"(공란)" 같이 표시하세요.

## 7. 헷갈리는 경계 사례 (FAQ)

UI 도움말 슬롯 → "?" 아이콘 클릭 시 모달

**Q1. 자유 문장도 `- ` 붙여 리스트로 해야 하나?**
A. 불필요. 단일 문단은 그대로 두세요. `- ` 는 **병렬 나열 2개 이상**일 때만.

**Q2. `- key:: value` 의 value로 긴 문장 넣어도 되나?**
A. OK. value 길이·형태 제한 없음. `- 휴직:: 소정의 사유로 일정기간동안 직원의 신분은 보유하나 직무에 종사하지 않음을 명함.` 같이 써도 됩니다.

**Q3. 일반 문장 vs `- key:: value` 결과 차이?**
A. 파이프라인 관점 차이는 **메타 필터 skip 여부** 하나. key-value는 "용어 = 정의"가 **명시적으로 결속**되어 인출이 더 정확합니다.

**Q4. 평문 메모에 `#` 를 해시태그처럼 써도 되나?**
A. 됩니다. 의미 처리 시 `#` 로 시작하는 줄이 있으면 카테고리 등록 후보가 되고, 없으면 평문으로 처리됩니다. v22 2차안은 모드 분리가 없으므로 사용자가 인지·선택할 필요 없습니다 — 시냅스가 알아서 흡수합니다.

**Q5. 원본에 `①②③` 같은 번호가 있으면?**
A. 번호는 **떼고** `- value` 리스트로 변환. 시냅스는 순서보다 의미 연결이 중요.

**Q6. 표(`|...|...|`)는 어떻게 처리되나?**
A. 현재 파이프라인은 표를 **자유 문장 묶음**으로 처리. 구조적 의미는 보존되지 않음. 표 내용이 중요하면 `- key:: value` 시리즈로 풀어 쓰는 걸 권장.
