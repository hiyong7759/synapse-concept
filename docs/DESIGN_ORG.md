# Synapse 설계 — 개인-조직 연결

## 기본 원칙

- 개인 그래프: 온디바이스. 완전히 사용자 소유.
- 조직 그래프: 조직 서버. 소속 시 접근 가능.
- 개인은 조직 그래프에 직접 접근 가능.
- 조직은 개인 그래프에 직접 접근 불가. 요청-응답 프로토콜을 통해서만 정보를 받는다.
- 개인은 여러 조직에 동시에 속할 수 있다.

---

## 구조

```
개인 앱 (클라이언트, 온디바이스)
  └── 단일 그래프 (내부 분리 없음)
  └── 오프라인 기본값
  └── 조직 A 연결 시: 조직 A 서버에 조회 요청 → 맥락 인출에 활용
  └── 조직 B 연결 시: 조직 B 서버에 조회 요청 → 맥락 인출에 활용

조직 서버 (자체 인프라, 설치형)
  └── 조직 그래프만 운영
  └── membership + role 기반 접근 제어
```

---

## 개인 → 조직 (조직 그래프 읽기)

```
개인 비서 → 조직 서버에 조회 요청
  ↓
조직 서버가 membership + role 확인
  ↓
권한 범위 내 노드/트리플 반환
  ↓
개인 비서가 인출/가공에 활용
```

---

## 조직 → 개인 (요청-응답 프로토콜)

조직은 개인 그래프에 직접 접근할 수 없다.

```
조직에서 요청 발송
  ↓
비서가 자동 수신 → 그래프 인출 → LLM 가공
  ↓
민감 정보 포함?
  No → 응답 초안 → 사용자 확인 → 전송
  Yes → 사용자에게 알림 → 진행/거부 → 전송
```

---

## 조직 그래프 캐시 정책

| 정보 유형 | 캐시 | 비고 |
|-----------|------|------|
| 민감 정보 | 불가 | 매번 서버 조회 |
| 비민감 정보 | 사용자 선택 | 기본값: 캐시 안 함 |

민감 판단: LLM 자동 판단 + 조직 관리자 태그.

---

## 조직 역할 모델

회사 관점의 5단계 역할. 운영 관리와 데이터 접근 권한을 통합한다.

| 역할 | 읽기 | 쓰기 | 멤버 관리 | 조직 삭제 | 접근 범위 |
|------|:----:|:----:|:---------:|:---------:|----------|
| admin | O | O | O | O | 전체 (= 구 host) |
| executive | O | O | X | X | 전략·영업비밀 포함 |
| hr | O | O | X | X | 인사·연봉 포함 |
| team_lead | O | O (팀 내) | X | X | 팀원 업무·성과 |
| employee | O | X | X | X | 본인 업무 정보만 |

**민감도 게이트 (Task 5A/5B-org):**
- 각 관계에 최소 열람 권한 태그 (`sensitive:personal_info:hr` 등)
- 질의자 권한이 최소 권한 미만이면 reject, 이상이면 confirm → 사용자 확인 후 열람

---

## 조직 연결 해제 (퇴사/탈퇴)

membership이 끊어지는 시점이 트리거.

```
조직과 연결 해제
  ↓
비서가 개인 이력으로 마이그레이션할 정보 정리
  - 참여 프로젝트명, 역할, 기간
  - 습득한 기술/스킬
  - 수행 업무 요약
  (조직 고유 정보 제외: 고객 정보, 계약 내용 등)
  ↓
사용자가 항목별 선택/수정
  ↓
선택된 정보 → 개인 그래프로 이관
  ↓
조직 캐시: 사용자에게 삭제/보관 선택
  ↓
조직 그래프 접근 차단
```

---

## 조직 서버 운영 모델

GitLab 모델. Synapse가 설치형 소프트웨어를 제공하고, 조직이 자체 서버에 운영한다.

- Synapse: 앱 + 서버 소프트웨어 개발/배포 + 초기 세팅 지원
- 조직: 서버 운영 주체 + 개인정보처리자
- Synapse 서버: 조직 그래프 미보유. 조직 운영에 개입하지 않음.
