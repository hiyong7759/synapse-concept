# Synapse 설계 — 개요

## 한 줄 정의

나의 맥락을 매번 설명하지 않아도 되는 세상.

개인과 조직의 정보를 지식 하이퍼그래프(여러 노드를 묶는 **지식 바구니**들의 집합)로 구조화하고, 어떤 상황에서든 필요한 맥락만 꺼내주는 엔진.

---

## 핵심 가치

사람은 누군가와 소통할 때마다 상대에 맞춰 자신의 맥락을 반복 설명해야 한다. 비서는 이 과정을 대신한다. 내 정보를 구조화해서 저장하고, 상황에 맞게 꺼내고, 필요하면 가공까지 해서 전달한다.

---

## 데이터 정책 (모든 설계의 전제)

Synapse는 사용자 데이터를 일절 수집하지 않는다. Synapse가 제공하는 것은 앱(인터페이스)과 서버 소프트웨어뿐이다.

- **개인 하이퍼그래프**: 사용자의 디바이스에만 존재. Synapse 서버에 전송되지 않음.
- **조직 하이퍼그래프**: 해당 조직의 서버에만 존재. Synapse 서버에 전송되지 않음.
- **Synapse 서버**: 앱 배포, 업데이트, 인증 등 서비스 운영 최소 정보만 처리. 하이퍼그래프 데이터 미보유.

Synapse는 데이터 관리 및 운영 주체가 아니다. 도구 제공자다.

### 외부 참조 데이터 정책

행정구역·표준 공공 데이터 같은 외부 참조는 **하이퍼그래프에 노드로 저장하지 않는다.** 필요 시점에 외부 API로 조회해서 응답 컨텍스트에 일회성으로 포함할 뿐. 매칭용 식별자도 노드에 부착하지 않는다. 개인 맥락 데이터는 온디바이스, 공공 참조는 외부 리소스 — 데이터 주권 원칙의 범위가 충돌하지 않는다.

### 지능체 분리 아키텍처

시냅스는 **저장 + 인출 엔진**이며 사고·합성은 외부 지능체가 수행한다.

- **외부 고급 LLM** (Claude Opus 등): 무거운 합성·요약·장문 생성
- **로컬 경량 LLM** (Gemma E2B): 분류·추론·쿼리 처리
- **사용자 본인 또는 협업자**: 직접 탐색·편집·승인

이 분리가 주는 성질:
- 시냅스는 LLM 없이도 독립 동작 (`--no-llm`)
- LLM 교체 가능, 특정 모델 비종속
- 개인 맥락은 시냅스 내부, 외부 LLM에는 질의 시 사용자가 선별 전달
- 잦은 작업은 로컬, 드문 고품질 작업만 외부 API

## 아키텍처 구조

```
개인 앱 (클라이언트)
  └── 단일 하이퍼그래프 (온디바이스)
  └── 온디바이스 LLM (MLX — api/mlx_server.py)
  └── 오프라인 기본값
  └── 조직 연결 시: 조직 서버에 조회 요청 → 맥락 인출에 활용

조직 서버 (자체 인프라)
  └── 조직 하이퍼그래프만 운영
  └── Synapse가 설치형 소프트웨어 제공 (GitLab 모델)
  └── 조직이 운영 주체 + 개인정보처리자

Synapse
  └── 앱 + 서버 소프트웨어 개발/배포
  └── 초기 세팅 지원
  └── 데이터 미보유, 개인정보처리자 아님
```

## 포지셔닝

| 구분 | 대표 서비스 | 한계 |
|------|-------------|------|
| 클라우드 AI 비서 | Jenova, SKT 에이닷 | 데이터가 서버에 저장됨 |
| 로컬 지식 관리 도구 | Obsidian, Logseq | 사용자가 직접 정리해야 함 |
| 온디바이스 AI 프레임워크 | OpenJarvis, Reor | RAG 기반. 관계 추론 불가. 미완성 서비스 |

**Synapse 차별점 4가지:**
1. 대화 기반 자동 노드 추출 — 말하면 문장·노드·문장 역참조가 자동 저장
2. 하이퍼엣지 기반 연결 — 선(엣지)이 아니라 문장·카테고리·별칭이라는 **지식 바구니**가 여러 노드를 동시에 묶는다. 의미 해석(인과·유사)은 외부 지능체 몫, 저장은 엔진 몫
3. 완전 로컬 — 온디바이스 모델 + 로컬 하이퍼그래프. 외부 LLM 없이도 독립 동작
4. 개인-조직 연결 프로토콜 — 별도 하이퍼그래프가 요청-응답으로 연결. 현재 없는 구조

### Obsidian과의 근본 차이

| | Obsidian | Synapse |
|---|---|---|
| 기본 단위 | 노트 (마크다운 파일) | 노드 (개념 단위) |
| 연결 | 사용자가 수동 [[wikilink]] | 자동: 같은 문장 바구니·같은 카테고리 바구니에 묶이면 연결 |
| 고립된 정보 | 링크 없으면 고립 | `node_mentions` + `node_categories`로 언급·분류만 되면 자연스럽게 도달 가능 |
| 인출 | 사용자가 직접 검색 | 비서가 탐색→필터→가공→응답 |
| 성격 | 사용자가 정리하는 도구 | 비서가 자동 저장하고 사용자가 검수·편집하는 서비스 |

## 하루 시나리오: 비서 없음 vs 있음

회사원 A씨, 38세. 오전 병원, 낮에 회사, 저녁 동호회.

| 시간 | 비서 없음 | 비서 있음 |
|------|-----------|-----------|
| 07:30 병원 | "이전에 어떤 약 드셨어요?" 기억 안 남. 갤러리 뒤짐. | "3월 처방 세레콕시브 200mg입니다. 처방전 이미지 있습니다." |
| 09:00 출근 | 지난 미팅 회의록 찾으러 메일/슬랙 10분 검색 | "지난 고객 미팅 핵심 요청사항 3건" 즉시 제공 |
| 12:00 점심 | 후배 이직 상담. 경력 머릿속으로 정리해서 구두 설명. 빠뜨림. | "내 경력 요약해줘" → 13년 경력, 주요 프로젝트 시간순 정리 |
| 18:00 동호회 | 회비 정산. 카톡 스크롤하며 입금 내역 30분 탐색 | "회비 정산 정리해줘" → 초안 생성. 확인 후 전송 |
| 20:00 이력서 | 기억나는 대로 적음. 빠진 게 있을 것 같음. | "어디에 지원하시나요?" → 맥락 파악 후 경력/성과 자동 인출. 초안 생성 |

**핵심 차이**: 비서 없으면 매 상황마다 직접 기억하고 찾고 정리하고 설명해야 한다. 비서 있으면 그 과정이 전부 생략된다.
