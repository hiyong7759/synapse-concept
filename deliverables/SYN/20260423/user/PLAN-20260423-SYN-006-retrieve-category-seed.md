## 1차 원인 가설 — 정확 매칭(`name = ?`) vs 실제 카테고리 이름 괴리

`engine/retrieve.py:133` `_match_start_categories`:
```python
rows = conn.execute(
    """WITH RECURSIVE sub(id) AS (
           SELECT id FROM categories WHERE name = ?
           ...
       )""",
    (kw,),
)
```

**exact match** 로 키워드와 `categories.name` 을 비교. 하지만 실제 DB 상태:

| 키워드 (retrieve-expand + split + Kiwi) | `categories.name` 실제값 | 매칭 결과 |
|---|---|---|
| `연차` | `제55조 (연차휴가의 사용 및 특별한 지정)` | ❌ (부분일치 불가) |
| `휴가` | `제6장 휴일 및 휴가` · `제49조 (가족돌봄 등을 위한 근로시간 단축)` | ❌ |
| `징계` | `제11장 표창 및 징계` · `제61조 (징계)` | ⭕ `제61조 (징계)` 는 exact? 아니, 괄호·번호 붙어있어서 ❌ |
| `수습기간` | `제7조 (수습기간)` | ❌ |
| `회사` | — (회사는 heading 에 등장 안 함) | ❌ |

즉 **문서 heading 이 "제N조 (주제)" 처럼 번호+괄호 포함 구조**라 자연어 키워드와 절대 매칭되지 않음. 반면 이전 PLAN-004 dogfood 에서 `건강`·`더나은` 같은 **짧은 단일 토큰 heading** 으로 테스트했을 때는 이 문제가 드러나지 않았다 — `_match_start_categories` 의 exact match 한계가 **문서 유형에 따라 침묵** 한 것.

## 2차 원인 가설 (배제 불필요 · M1 조사 대상)

- **retrieve-expand 가 카테고리명 자체를 키워드로 제안하지 않음** — 확장 키워드가 자연어 단어 위주이고 "제6장 휴일 및 휴가" 같은 표제는 제안 안 됨
- **categories 테이블에 alias 컬럼 없음** — 동의어 매칭 불가
- **Kiwi 가 카테고리명에 적용되지 않음** — 저장 시 `제6장 휴일 및 휴가` 를 토큰화해두지 않으므로 역인덱스 없음

## 해결 방향 후보 (M1 결정)

### A. LIKE 부분일치 (간단)
```python
WHERE name LIKE '%' || ? || '%'
```
- 장점: 1 줄 수정. `연차` → `제55조 (연차휴가…)` 매칭됨
- 단점: 너무 관대함. `휴` 키워드 → `휴가`·`휴일`·`휴직` 전부 매칭 → 시드 폭발. `.`·`(`·`)` 같은 문자 끼어있으면 오매칭 가능.

### B. 카테고리명 토큰화 + 역인덱스 테이블 (정교)
- 신규 `category_tokens(category_id, token)` 테이블
- 저장 시 Kiwi 로 `categories.name` 토큰화 → 역인덱스 적재
- 인출 시 키워드 → token 매칭 → category_id
- 장점: 노드 추출과 동일한 품질. 형태소 기반.
- 단점: 스키마 변경. 저장 파이프라인에도 추가.

### C. aliases 확장 (카테고리 별칭)
- `categories` 에 alias 추가 허용 (사용자 또는 AI 워커)
- 시드 매칭 시 name + aliases 둘 다 조회
- 장점: 구조적. "연차" → category `제55조` alias 로 등록 가능.
- 단점: 저장 시점 어떻게 자동 채울지 별도 문제 (결국 B 와 비슷)

### D. 재현 데이터 DB 축소·노드 확장 재평가
- 취업규칙 문서는 **예외적으로 긴 heading** 을 갖는 케이스일 수 있음
- 일상 사용자 저장은 `#건강` · `#직장.더나은` 처럼 짧은 토큰
- 실 사용 패턴에서는 이 버그가 잘 드러나지 않을 수도
- 단점: "해결 안 함" 에 가까움. 리스크 숨김.
